Role / Permission / Admin Management — Implementation Plan

Adds Spatie Laravel Permission v7 for role + permission management, a seeded permission catalog organized by group, role status (Active / Deactive), and a full Admin Management CRUD — all gated by Spatie\Permission\Middleware\PermissionMiddleware at the route level. Builds on top of the existing admin auth foundation (see admin-auth-plan.html).

Laravel 13 PHP 8.4 Sanctum 4 spatie/laravel-permission ^7.4 PHPUnit 12

1 Goals & Scope

  • Install + bootstrap Spatie Laravel Permission v7 on the existing User model (single shared sanctum guard).
  • Extend the published Spatie schema with two custom columns:
    • permissions.group — string, for UI grouping (e.g. Roles, Admins, Users).
    • roles.status — tinyInteger backed by a RoleStatusEnum (Active / Deactive), mirroring the existing UserStatusEnum pattern.
  • Seed the full permission catalog via a dedicated PermissionSeeder, grouped by feature area (idempotent — safe to re-run after each release).
  • Role management endpoints: list, create, show, update, delete — create assigns a name, status, and a set of permissions.
  • Admin management endpoints: list, create, show, update, delete — each admin gets exactly one role.
  • Hard self-protection rules:
    • An admin cannot edit or delete their own admin record.
    • An admin cannot edit or delete the role currently assigned to themselves.
  • Every role is treated equally — there is no "Super Admin" bypass, no hard-coded role name, no Gate::before shortcut. Access to any feature is determined only by the permissions assigned to the role(s) the admin holds. To grant an admin "full access," assign a role that has every permission attached — nothing more, nothing less.
  • Every protected route is gated by permission:<name> middleware — never relying on controller-side $user->can() as the only line of defense.
  • 100% test coverage of happy + failure + edge paths for every endpoint.

6 Permission Catalog (Seeder)

Database\Seeders\PermissionSeeder new

Permission names & their groups are written directly in the seeder as a plain array — no enum indirection. Add a new group / permission here when a new feature lands, then re-run the seeder.

class PermissionSeeder extends Seeder
{
    use WithoutModelEvents;

    public function run(): void
    {
        $groups = [
            'Roles' => [
                'roles.view',
                'roles.create',
                'roles.update',
                'roles.delete',
            ],
            'Admins' => [
                'admins.view',
                'admins.create',
                'admins.update',
                'admins.delete',
            ],
            'Users' => [
                'users.view',
                'users.update',
            ],
            // add a new group / permissions here as features grow
        ];

        foreach ($groups as $group => $names) {
            foreach ($names as $name) {
                Permission::updateOrCreate(
                    ['name' => $name, 'guard_name' => 'sanctum'],
                    ['group' => $group],
                );
            }
        }

        app(PermissionRegistrar::class)->forgetCachedPermissions();
    }
}

Idempotent — re-running the seeder after a release picks up any newly added rows and updates the group on existing ones without duplicating.

Database\Seeders\RoleSeeder new

Seeds one Administrator role with every permission attached — purely so the default seeded admin has a working starting point. This role has no special powers in code; it is just an ordinary role whose row in role_has_permissions happens to cover the whole catalog. Once the system is running, an admin with roles.update can rename it, remove permissions, or delete it like any other role.

class RoleSeeder extends Seeder
{
    use WithoutModelEvents;

    public function run(): void
    {
        $role = Role::updateOrCreate(
            ['name' => 'Administrator', 'guard_name' => 'sanctum'],
            ['status' => RoleStatusEnum::Active],
        );

        // Attach every seeded permission. Re-runs stay in sync.
        $role->syncPermissions(Permission::all());

        app(PermissionRegistrar::class)->forgetCachedPermissions();
    }
}

Database\Seeders\DatabaseSeeder edit

Call the new seeders in order, then assign the seeded admin to the Administrator role:

$this->call([PermissionSeeder::class, RoleSeeder::class]);

10 Self-Protection Rules

Enforced in the service layer (so the rule holds regardless of which controller path reaches it). Controllers translate the exception to an HTTP response.

ActionBlocked when…StatusException
Update admin user$target->id === $request->user()->id403CannotModifySelfException
Delete admin user$target->id === $request->user()->id403CannotModifySelfException
Update role$request->user()->hasRole($role->name)403CannotModifyOwnRoleException
Delete role$request->user()->hasRole($role->name)403CannotModifyOwnRoleException
Delete rolerole is attached to any user (count > 0)422RoleStillInUseException

"Cannot edit own role" applies to the role the current admin is assigned to. They can still create new roles and assign other admins to different roles. The intent is: don't let an admin lock themselves out by stripping permissions from their own role mid-session.

"Cannot delete a role still in use" — a role can be deleted only after every admin assigned to it has been moved to a different role (or removed). The service performs $role->users()->count() before delete; if non-zero, it throws RoleStillInUseException → 422 with the message "This role is currently assigned to N admin(s). Reassign them before deleting."

12 Controller / Service Sketches

RoleController (selected)

public function store(StoreRoleRequest $request): JsonResponse
{
    $role = $this->roleService->create($request->validated());
    return $this->success(data: RoleResource::make($role), message: 'Role created.', status: 201);
}

public function update(UpdateRoleRequest $request, Role $role): JsonResponse
{
    try {
        $role = $this->roleService->update($role, $request->validated(), actor: $request->user());
    } catch (CannotModifyOwnRoleException $e) {
        return $this->error($e->getMessage(), 403);
    }
    return $this->success(data: RoleResource::make($role->load('permissions')), message: 'Role updated.');
}

RoleService (rules in one place)

public function update(Role $role, array $data, User $actor): Role
{
    if ($actor->hasRole($role->name)) {
        throw new CannotModifyOwnRoleException(...);
    }

    DB::transaction(function () use ($role, $data) {
        $role->fill(['name' => $data['name'], 'status' => $data['status']])->save();
        $role->syncPermissions(Permission::whereIn('id', $data['permissions'])->get());
    });

    app(PermissionRegistrar::class)->forgetCachedPermissions();
    return $role->refresh();
}

AdminUserService (delete)

public function delete(User $target, User $actor): void
{
    if ($target->id === $actor->id) {
        throw new CannotModifySelfException(...);
    }

    $target->tokens()->delete();   // revoke any sessions the target had
    $target->delete();
}

Implementation notes

  • Eager-load before resolve: in AdminAuthService::login() and ProfileController::show(), call $user->load('roles.permissions') before passing the model to AdminResource::make() — avoids N+1 inside the resource.
  • Roles shape: emit only name and status_label per role — the frontend never needs the role id or the raw integer status for session-level checks. Built inline in AdminResource::toArray(): 
    'roles' => $this->roles->map(fn (Role $role) => [
    'name'         => $role->name,
    'status_label' => $role->status->label(),
])->values(),
  • Permissions shape: a single flat array of permission names — not grouped, no group field. Easy to includes()-check on the client. 
    'permissions' => $this->getAllPermissions()->pluck('name')->values(),
    $user->getAllPermissions() (Spatie helper) returns the union of permissions across all assigned roles + any direct permissions. Same code path for every admin — no role is special-cased.
  • Single source of truth: Both endpoints return the same AdminResource shape. The frontend persists it on login and re-fetches via GET /profile on app boot / role change.

17 Confirmed Decisions

All four points below are locked in — the implementation must follow these rules. Re-open only with explicit approval.

  • ✓ Single role per admin. Each admin is assigned exactly one role. Request payloads use scalar role_id (not an array). Persisted via $user->syncRoles([$role]) — syncRoles (not assignRole) so that updating an admin's role replaces the previous one atomically.
  • ✓ Hard delete admins (no SoftDeletes). $target->tokens()->delete() then $target->delete(). No deleted_at column, no SoftDeletes trait on the User model.
  • ✓ Deactive role behavior. A role with status=Deactive keeps working for every admin already assigned to it — no automatic revocation, no permission stripping.
