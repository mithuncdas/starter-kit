Admin Authentication — Implementation Plan

Sanctum token-based auth on the User model with user_type + status enums, scoped under an Admin namespace and /api/admin URL prefix. Single shared sanctum guard — admin vs. user is enforced via user_type + a thin admin middleware.

1 Goals & Scope

  • Single users table for both Admin and regular User accounts, distinguished via user_type.
  • Token-based authentication via Laravel Sanctum (HasApiTokens).
  • All admin endpoints live under /api/admin/* and use the Admin\ namespace for controllers, requests, services, resources, notifications.
  • Endpoints: login, logout, forgot password, reset password, profile view, profile update, change password.
  • Reset-password & change-password flows must revoke all other Sanctum tokens (logout other devices).
  • Rate limiting on login, forgot password, and reset password.
  • 100% test coverage of happy + failure + edge paths for every endpoint.

5 Auth Config & Middleware

No separate admin guard. Both users and admins authenticate via the same sanctum guard. Admin-only routes are gated by a custom admin middleware that checks the authenticated user's user_type and status.

// app/Http/Middleware/EnsureAdmin.php
if (! $request->user() || ! $request->user()->isAdmin() || ! $request->user()->isActive()) {
    abort(403, 'Admin access required.');
}

Registered as the alias admin in bootstrap/app.php. Applied after auth:sanctum on all admin-protected routes:

Route::middleware(['auth:sanctum', 'admin'])->group(...);

Password reset is OTP-based (6-digit, 5-minute expiry) — Laravel's password_reset_tokens table and password broker are not used for admin reset. See §6a for the OTP model/table and §9 for the flow.

5b API Response Convention

Trait: App\Http\Responses\ApiResponder

All controllers use ApiResponder; and call $this->success(...) / $this->error(...). Never build a raw response()->json(...) in a controller.

// Success
$this->success(data: $resourceOrArray, message: 'OK', status: 200): JsonResponse
// Error
$this->error(message: 'Something went wrong', status: 400, errors: $detailsOrNull): JsonResponse

Standard response shape:

// 2xx
{ "success": true,  "message": "OK",         "data": {...} }
// 4xx / 5xx
{ "success": false, "message": "Bad input",  "errors":  {...} | null }

Resources are mandatory

  • Any model (or collection) returned to the client must be wrapped in an API Resource — never expose Eloquent models directly.
  • Admin endpoints use App\Http\Resources\Admin\AdminResource for the authenticated admin.
  • The login endpoint returns ['token' => $plain, 'admin' => AdminResource::make($user)->resolve()] via $this->success().

5a OTP Storage (generic)

Model: App\Models\Otp  ·  Table: otps

Generic OTP store supporting email and phone channels. The admin password-reset flow always uses the email column for now; the phone column is in place so future flows (e.g. user phone verification, 2FA) can reuse the same table and service without a schema migration.

ColumnTypeNotes
idbigint PKauto-increment
emailstring, nullable, indexedUsed by email-channel OTPs (e.g. admin password reset)
phonestring, nullable, indexedReserved for future SMS OTP flows; nullable for now
otpstringHashed (bcrypt) — never store plaintext OTPs
expires_attimestampcreated_at + 5 min
created_at / updated_attimestampsstandard

DB-level CHECK that at least one of email/phone is non-null would be ideal; for portability we'll enforce it in the service layer instead.

Service: App\Services\OtpService

Generic — accepts a channel + identifier. Two channel constants: OtpService::CHANNEL_EMAIL, OtpService::CHANNEL_PHONE.

OtpService::issue(channel: 'email', identifier: $email): string  // returns plaintext OTP
OtpService::verify(channel: 'email', identifier: $email, otp: $otp): bool
OtpService::purge(channel: 'email', identifier: $email): void   // delete ALL rows for that contact

Rules

  • Issue: first delete expired rows for that (channel, identifier); if a non-expired row still exists, throw → controller responds 422 (“An OTP was already sent — please wait until it expires.”).
  • Verify: find the latest row for (channel, identifier) where expires_at > now(), then Hash::check() the supplied OTP. On success: caller invokes purge() to delete ALL rows for that contact, updates password, and revokes ALL Sanctum tokens.
  • Plaintext OTP is generated server-side, hashed before insert, and sent via the appropriate channel (Notification for email today, SMS gateway later); plaintext is never persisted.

8 Rate Limiting Strategy

Defined in App\Providers\AppServiceProvider::boot() using RateLimiter::for():

  • admin-login — 5/min, keyed by ip|email.
  • admin-forgot — 3/min, keyed by ip|email (the OTP "already sent" check is the second layer that prevents abuse within the 5-min window).
  • admin-reset — 5/min, keyed by ip|email (prevents OTP brute-force).

Applied via throttle:<name> route middleware.

9 Flows

Login

POST /api/admin/login { email, password, device_name? } │ ├─▶ throttle:admin-login ├─▶ LoginRequest validates ├─▶ AdminAuthService::login() │ ├─ find user where email AND user_type=Admin (any status) │ ├─ if !user OR !Hash::check(password) → 422 "Invalid credentials" │ ├─ if user->status == Deactive → 403 "Account is deactivated, please contact administrator" │ └─ $user->createToken($device_name ?? 'admin') └─▶ 200 success({ token, admin: AdminResource })

Forgot password (issue OTP)

POST /api/admin/forgot-password { email } │ ├─▶ throttle:admin-forgot ├─▶ Look up admin user (user_type=Admin, any status) │ ├─ not found → 422 "No admin account found for this email" │ └─ user->status == Deactive → 403 "Account is deactivated, please contact administrator" ├─▶ OtpService::issue(channel: 'email', identifier: $email) │ ├─ DELETE FROM otps WHERE email=? AND expires_at <= now() // cleanup expired │ ├─ if row exists where expires_at > now() → throw → controller returns 422 "OTP already sent, wait until it expires" │ ├─ $plain = random 6-digit string (e.g. "739204") │ ├─ INSERT { email, phone=null, otp = Hash::make($plain), expires_at = now()+5min } │ └─ return $plain ├─▶ Notification::route('mail', $email)->notify(new AdminPasswordResetOtpNotification($plain)) └─▶ 200 success(message: "OTP sent to your email")

Per product decision, we trade enumeration protection for UX clarity: explicit messages for unknown / deactivated accounts.

Reset password (verify OTP)

POST /api/admin/reset-password { email, otp, password, password_confirmation } │ ├─▶ throttle:admin-reset ├─▶ Look up admin user (user_type=Admin, status=Active) — else 422 ├─▶ OtpService::verify(channel: 'email', identifier: $email, otp: $otp) │ ├─ row = otps WHERE email=? AND expires_at > now() ORDER BY id DESC LIMIT 1 │ ├─ if !row OR !Hash::check($otp, $row->otp) → throw → 422 "Invalid or expired OTP" │ └─ return true ├─▶ $user->update(['password' => Hash::make($password)]) ├─▶ OtpService::purge(channel: 'email', identifier: $email) ← delete ALL OTPs ├─▶ $user->tokens()->delete() ← LOG OUT ALL DEVICES └─▶ 200 { message: "Password reset successful" }

Change password (authenticated)

POST /api/admin/change-password { current_password, password, password_confirmation } │ ├─▶ auth:sanctum + admin ├─▶ ChangePasswordRequest validates (with `current_password` rule) ├─▶ AdminAuthService::changePassword() │ ├─ update password │ └─ $admin->tokens()->where('id','!=',currentTokenId)->delete() │ ← KEEP current token, revoke others └─▶ 200 { message: "Password changed" }

Profile view / update

GET /api/admin/profile → AdminResource PUT /api/admin/profile { name, email } → validates unique email (ignore self) → returns AdminResource

Logout

POST /api/admin/logout (auth:sanctum + admin) └─▶ $request->user()->currentAccessToken()->delete() └─▶ 200 { message: "Logged out" }
