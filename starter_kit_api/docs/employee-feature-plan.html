2. Goals & Use Cases

Every admin (any user_type = Admin) is automatically an Employee of the company. Regular users are not employees. The Employee record carries HR data; the User record continues to carry identity/auth data. The split must let us:

  • Know who reports to whom (manager ↔ reports).
  • Group employees by department / designation for org-chart and reporting.
  • Track lifecycle: hired → active → on-leave → terminated.
  • Generate an employee code (EMP-0001…) that we expose in URLs and reports without leaking user IDs.
  • Hang future HR tables (leaves, attendances, payslips…) off a single employee_id FK.
  • Keep regular-user flows untouched — no HR clutter on the customer-facing side.

3. Design Options & Trade-offs

OptionShapeProsCons
A. Add HR columns to users One fat users table Zero joins. No new model. Nullable HR columns for every regular user. Identity table grows unbounded. Every HR-only migration touches the auth table. Bad blast radius.
B. Separate employees table, 1:1 with users employees.user_id unique FK HR columns isolated. Future HR tables FK to employees.id, not users.id. Regular users are unaffected. Cleanest evolution path. Two-row create on admin signup (mitigated by service + DB transaction).
C. STI / polymorphic “profile” per user type customer_profiles, employee_profiles via profile_type/profile_id Most extensible long-term. Premature for this codebase. Adds complexity (polymorphic joins, eager loading gotchas) for a benefit we don’t yet need.
Decision driver: HR is a separate domain with its own lifecycle and reports. Mixing it into users (Option A) is the move that’s cheap today and expensive forever — every future HR migration becomes a high-blast-radius auth-table migration. Option B is the same cost today and isolates the blast radius.

4. Recommended Approach

Go with Option B — a dedicated employees table, 1:1 with users:

  • Every User with user_type = Admin gets exactly one Employee row.
  • Regular users (UserTypeEnum::User) do not get an Employee row.
  • users.user_type remains the source of truth for “is this account an admin?” (auth + middleware unchanged).
  • The Employee row is created/updated inside AdminUserService in the same DB transaction as the admin User. This guarantees an admin always has an employee row — no observer side-effects, no orphan rows.
  • All future HR tables (leaves, attendances, payslips, employee_documents) FK to employees.id, not users.id.
  • Address stays on users (see §8 for the rationale).

7.1 Employee model

app/Models/Employee.php
namespace App\Models;

use App\Enums\EmployeeStatusEnum;
use App\Enums\EmploymentTypeEnum;
use App\Enums\GenderEnum;
use Database\Factories\EmployeeFactory;
use Illuminate\Database\Eloquent\Attributes\Fillable;
use Illuminate\Database\Eloquent\Builder;
use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;

#[Fillable([
    'user_id', 'employee_code', 'designation', 'department',
    'employment_type', 'status', 'joining_date', 'end_date',
    'phone', 'date_of_birth', 'gender',
    'emergency_contact_name', 'emergency_contact_phone', 'notes',
])]
class Employee extends Model
{
    /** @use HasFactory<EmployeeFactory> */
    use HasFactory;

    protected function casts(): array
    {
        return [
            'employment_type' => EmploymentTypeEnum::class,
            'status'          => EmployeeStatusEnum::class,
            'gender'          => GenderEnum::class,
            'joining_date'    => 'date',
            'end_date'        => 'date',
            'date_of_birth'   => 'date',
        ];
    }

    /** @return BelongsTo<User, $this> */
    public function user(): BelongsTo
    {
        return $this->belongsTo(User::class);
    }

    /** @param  Builder<self>  $query */
    public function scopeActive(Builder $query): void
    {
        $query->where('status', EmployeeStatusEnum::Active);
    }

    /**
     * @param  Builder<self>  $query
     * @param  array<string, mixed>  $filters
     */
    public function scopeFilter(Builder $query, array $filters): void
    {
        if (! empty($filters['search'])) {
            $search = $filters['search'];
            $query->where(function (Builder $q) use ($search): void {
                $q->where('employee_code', 'like', "%{$search}%")
                  ->orWhere('designation', 'like', "%{$search}%")
                  ->orWhere('department', 'like', "%{$search}%")
                  ->orWhereHas('user', fn (Builder $u) =>
                      $u->where('name', 'like', "%{$search}%")
                        ->orWhere('email', 'like', "%{$search}%"));
            });
        }

        if (! empty($filters['department'])) {
            $query->where('department', $filters['department']);
        }

        if (isset($filters['status']) && $filters['status'] !== '') {
            $query->where('status', (int) $filters['status']);
        }
    }
}

8. Where Address Lives (and other “shared” data)

The existing user_addresses table is keyed on user_id — not employee_id. The right move is to keep it that way. Reasoning:

ConcernStays on UserMoves to Employee
Identity (name, email, password)✓
Auth status (users.status)✓
Roles & permissions (Spatie)✓ (morphs to users)
Address(es)✓ — works for regular users too; primary address is “where this person is”, not “where they work”
HR profile fields (designation, department, manager, joining_date)✓
HR lifecycle (status, end_date)✓
HR sub-modules (leaves, attendance, payslips, documents)✓ (FK employee_id)

Reading an employee’s primary address becomes one extra hop:

$employee = Employee::with('user.primaryAddress.adminArea.ancestorsAndSelf.level')
    ->findOrFail($id);

$address = $employee->user->primaryAddress;
$division = $address->adminArea->ancestorsAndSelf
    ->firstWhere('level.code', 'DIVISION');
If, later, HR needs a distinct “work address” (office location), add it as a new column or table on employees (e.g. employees.office_admin_area_id or an employee_work_locations pivot). Don’t retrofit user_addresses to mean two things.

9. Service Layer Changes

Today AdminUserService::create() creates a User + syncs a role. We extend it to also create the Employee in the same transaction, so an admin row never exists without its employee row.

9.1 AdminUserService::create (revised)

app/Services/Admin/AdminUserService.php (diff)
public function create(array $data): User
{
    return DB::transaction(function () use ($data): User {
        $user = User::query()->create([
            'name'              => $data['name'],
            'email'             => $data['email'],
            'password'          => Hash::make($data['password']),
            'user_type'         => UserTypeEnum::Admin,
            'status'            => UserStatusEnum::from($data['status']),
            'email_verified_at' => now(),
        ]);

        $role = Role::query()->findOrFail($data['role_id']);
        $user->syncRoles([$role]);

        // NEW: create the matching Employee row.
        $this->employees->createFor($user, $data);

        return $user->load(['roles.permissions', 'permissions', 'employee']);
    });
}

9.2 New EmployeeService

app/Services/Admin/EmployeeService.php
namespace App\Services\Admin;

use App\Enums\EmployeeStatusEnum;
use App\Enums\EmploymentTypeEnum;
use App\Models\Employee;
use App\Models\User;
use Illuminate\Support\Facades\DB;

class EmployeeService
{
    /**
     * Called from AdminUserService during admin creation.
     *
     * @param  array{
     *   designation?: string|null,
     *   department?: string|null,
     *   employment_type?: int,
     *   joining_date?: string,
     *   phone?: string|null,
     * }  $data
     */
    public function createFor(User $user, array $data): Employee
    {
        return $user->employee()->create([
            'employee_code'   => $this->nextCode(),
            'designation'     => $data['designation']        ?? null,
            'department'      => $data['department']         ?? null,
            'employment_type' => EmploymentTypeEnum::from($data['employment_type'] ?? EmploymentTypeEnum::FullTime->value),
            'status'          => EmployeeStatusEnum::Active,
            'joining_date'    => $data['joining_date']       ?? now()->toDateString(),
            'phone'           => $data['phone']              ?? null,
        ]);
    }

    public function update(Employee $employee, array $data): Employee
    {
        return DB::transaction(function () use ($employee, $data): Employee {
            $employee->fill($data)->save();
            return $employee->fresh(['user']);
        });
    }

    public function terminate(Employee $employee, string $endDate): Employee
    {
        return DB::transaction(function () use ($employee, $endDate): Employee {
            $employee->forceFill([
                'status'   => EmployeeStatusEnum::Terminated,
                'end_date' => $endDate,
            ])->save();

            // Also lock the auth side.
            $employee->user->forceFill(['status' => \App\Enums\UserStatusEnum::Inactive])->save();
            $employee->user->tokens()->delete();

            return $employee->fresh(['user']);
        });
    }

    /** Generates the next EMP-XXXX code. Race-safe via DB lock. */
    protected function nextCode(): string
    {
        return DB::transaction(function (): string {
            $last = Employee::query()->lockForUpdate()->orderByDesc('id')->value('employee_code');
            $n = $last ? ((int) substr($last, 4)) + 1 : 1;
            return sprintf('EMP-%04d', $n);
        });
    }
}
Inject EmployeeService into AdminUserService via constructor: public function __construct(protected EmployeeService $employees) {}. Both services live under App\Services\Admin, matching the existing convention.

13. Backfill & Seeders

Existing admin rows (the seeded admin@example.com in DatabaseSeeder) won’t have employees rows after migration. Two options:

  1. Migration-time backfill: append a DB::table('users') + DB::table('employees')->insert() loop in the up() method of the create migration, generating codes EMP-0001….
  2. Idempotent seeder: add EmployeeSeeder to DatabaseSeeder::run() that iterates all User::admins()->whereDoesntHave('employee') and creates an Employee row each. Re-runnable.

Option 2 is preferred — the create migration stays schema-only, and the seeder doubles as the “fix orphan admins” tool if anyone ever sneaks past AdminUserService.

database/seeders/EmployeeSeeder.php
namespace Database\Seeders;

use App\Models\User;
use App\Services\Admin\EmployeeService;
use Illuminate\Database\Seeder;

class EmployeeSeeder extends Seeder
{
    public function run(EmployeeService $employees): void
    {
        User::query()
            ->admins()
            ->whereDoesntHave('employee')
            ->get()
            ->each(fn (User $u) => $employees->createFor($u, [
                'designation' => 'Administrator',
                'department'  => 'Operations',
            ]));
    }
}

17. Risks & Open Questions

  • Two statuses (users.status vs employees.status). They mean different things (auth lock vs HR lifecycle). Document this and keep the terminate action as the canonical way to put them in sync. Don’t add a synced trigger — explicit beats clever.
  • Employee code collisions under high concurrency. The lockForUpdate() approach in nextCode() serializes admin creation. If that ever becomes a bottleneck, switch to a DB sequence (Postgres) or a numbers table with row-level locks.
  • Sensitive data exposure. Until employees.view-sensitive ships, do not return DOB / emergency contacts / notes in EmployeeResource. Default to hidden; opt-in via the permission.
  • What about non-admin employees? Out of scope. If the company ever has employees who don’t log in (e.g. blue-collar staff), revisit by relaxing the users.user_type = Admin constraint and allowing “employee without account” via a nullable employees.user_id. Defer until there’s a real ask.
  • Soft deletes? Don’t add them now. terminate + end_date capture the “left the company” state without the soft-delete complexity.
