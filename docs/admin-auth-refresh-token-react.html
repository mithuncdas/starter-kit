1. Architecture overview

The API issues two tokens on a successful login. They have different lifetimes, different storage locations, and different privileges:

Access token in memory

Short-lived (10 min). Returned in the JSON body. Stored only in React state / a JS variable. Sent on every API call in the Authorization: Bearer header. Grants the access ability.

Refresh token httpOnly cookie

Longer-lived (60 min, sliding). Delivered as an httpOnly, Secure, SameSite=Strict cookie. JavaScript cannot read it. Only ever sent to /api/admin/*. Grants the refresh ability only.

Why this split? Keeping the access token out of localStorage means an XSS payload can't exfiltrate a long-lived credential — the in-memory access token dies in 10 minutes and vanishes on reload. The refresh token is long-lived, but it lives in an httpOnly cookie that JavaScript physically cannot read, so XSS can't steal it either.

4. Token lifecycle

  1. Login → store access_token in memory; the browser silently stores the refresh cookie.
  2. Authenticated calls → attach Authorization: Bearer <access>.
  3. Access token expires (after 10 min) → next call returns 401.
  4. Silent refresh → SPA calls /refresh; cookie goes automatically; gets a new access token and a rotated refresh cookie.
  5. Retry → the original failed request is replayed with the new access token. The user notices nothing.
  6. Refresh token expires (after 60 min idle) → /refresh returns 401 → redirect to login.
  7. Logout → both tokens revoked, cookie cleared.
Sliding window: every successful /refresh issues a brand-new refresh token valid for another 60 minutes. So an actively-used session never forces a re-login — only 60 minutes of inactivity does.

5. React — axios instance

The single most important line is withCredentials: true. Without it the browser will not send (or store) the refresh cookie, and every refresh attempt fails.

src/lib/api.js 
import axios from "axios";

export const api = axios.create({
  baseURL: "/api",            // or "https://api.example.com/api"
  withCredentials: true,      // REQUIRED — sends & receives the httpOnly refresh cookie
  headers: { "Accept": "application/json" },
});
Cross-origin? If the SPA and API are on different origins, the API's CORS config must set supports_credentials => true and list the exact SPA origin (a wildcard * is not allowed together with credentials).

6. React — auth context (access token in memory)

We hold the access token in two places: React state (so components re-render on login/logout) and a plain module variable (so the axios interceptor can read the latest value synchronously without React).

src/auth/token-store.js 
// A non-React holder so interceptors can read/write the token outside the React tree.
let accessToken = null;

export const getAccessToken = () => accessToken;
export const setAccessToken = (t) => { accessToken = t; };
export const clearAccessToken = () => { accessToken = null; };
src/auth/AuthContext.jsx 
import { createContext, useContext, useEffect, useState, useCallback } from "react";
import { api } from "../lib/api";
import { setAccessToken, clearAccessToken } from "./token-store";

const AuthContext = createContext(null);

export function AuthProvider({ children }) {
  const [admin, setAdmin]       = useState(null);
  const [booting, setBooting]   = useState(true);  // true until the boot refresh resolves

  // Save the access token both in memory (for axios) and in state (for the UI).
  const applySession = useCallback((data) => {
    setAccessToken(data.access_token);
    setAdmin(data.admin);
  }, []);

  const login = useCallback(async (email, password) => {
    const { data } = await api.post("/admin/login", { email, password });
    applySession(data.data);
    return data.data.admin;
  }, [applySession]);

  const logout = useCallback(async () => {
    try { await api.post("/admin/logout"); } catch (_) { /* ignore */ }
    clearAccessToken();
    setAdmin(null);
  }, []);

  // On first mount, try to restore the session from the refresh cookie.
  useEffect(() => {
    setBootRefreshHandlers(applySession, () => setAdmin(null));   // see interceptor file
    api.post("/admin/refresh")
      .then(({ data }) => applySession(data.data))
      .catch(() => { clearAccessToken(); setAdmin(null); })
      .finally(() => setBooting(false));
  }, [applySession]);

  return (
    <AuthContext.Provider value={{ admin, booting, login, logout, isAuthenticated: !!admin }}>
      {children}
    </AuthContext.Provider>
  );
}

export const useAuth = () => useContext(AuthContext);

7. React — login screen

src/pages/Login.jsx 
import { useState } from "react";
import { useAuth } from "../auth/AuthContext";
import { useNavigate } from "react-router-dom";

export default function Login() {
  const { login } = useAuth();
  const navigate = useNavigate();
  const [form, setForm]   = useState({ email: "", password: "" });
  const [error, setError] = useState(null);
  const [busy, setBusy]   = useState(false);

  const onSubmit = async (e) => {
    e.preventDefault();
    setBusy(true); setError(null);
    try {
      await login(form.email, form.password);
      navigate("/dashboard");
    } catch (err) {
      setError(err?.response?.data?.message ?? "Login failed.");
    } finally {
      setBusy(false);
    }
  };

  return (
    <form onSubmit={onSubmit}>
      {error && <p className="error">{error}</p>}
      <input type="email"    value={form.email}    onChange={e => setForm({ ...form, email: e.target.value })} />
      <input type="password" value={form.password} onChange={e => setForm({ ...form, password: e.target.value })} />
      <button disabled={busy}>{busy ? "Signing in…" : "Sign in"}</button>
    </form>
  );
}
Notice there is nothing about the refresh token here. You never touch it — the browser stores the Set-Cookie automatically, and only ever sends it back to /api/admin/*.

8. React — silent refresh interceptor

This is the heart of the system. Two axios interceptors:

  • Request interceptor — attaches the current in-memory access token to every outgoing call.
  • Response interceptor — catches 401, calls /refresh once, then replays the failed request. Concurrent 401s are queued so only one refresh happens.
src/lib/interceptors.js 
import { api } from "./api";
import { getAccessToken, setAccessToken, clearAccessToken } from "../auth/token-store";

// Handlers wired up by AuthProvider so a successful boot/refresh updates React state too.
let onRefreshed = () => {};
let onRefreshFailed = () => {};
export function setBootRefreshHandlers(refreshed, failed) {
  onRefreshed = refreshed; onRefreshFailed = failed;
}

// ---- Request: attach the access token ----
api.interceptors.request.use((config) => {
  const token = getAccessToken();
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// ---- Response: refresh-on-401, with a single-flight queue ----
let isRefreshing = false;
let queue = [];   // requests that arrived while a refresh was already in flight

const flushQueue = (error, token = null) => {
  queue.forEach(({ resolve, reject }) => (error ? reject(error) : resolve(token)));
  queue = [];
};

api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const original = error.config;
    const status   = error.response?.status;

    // Only handle a genuine 401, never retry the refresh call itself, retry each request once.
    const isRefreshCall = original.url?.includes("/admin/refresh");
    if (status !== 401 || original._retried || isRefreshCall) {
      return Promise.reject(error);
    }
    original._retried = true;

    // If a refresh is already running, wait for it instead of starting another.
    if (isRefreshing) {
      return new Promise((resolve, reject) => queue.push({ resolve, reject }))
        .then((token) => {
          original.headers.Authorization = `Bearer ${token}`;
          return api(original);
        });
    }

    isRefreshing = true;
    try {
      const { data } = await api.post("/admin/refresh");   // cookie sent automatically
      const newToken = data.data.access_token;
      setAccessToken(newToken);
      onRefreshed(data.data);                // keep React state in sync
      flushQueue(null, newToken);               // release queued requests
      original.headers.Authorization = `Bearer ${newToken}`;
      return api(original);                    // replay the original request
    } catch (refreshError) {
      flushQueue(refreshError, null);
      clearAccessToken();
      onRefreshFailed();                       // e.g. clear admin -> route guard sends to /login
      return Promise.reject(refreshError);
    } finally {
      isRefreshing = false;
    }
  }
);

Import this file once, early (e.g. in main.jsx) so the interceptors register before any request is made:

src/main.jsx 
import "./lib/interceptors";   // side-effect import — registers the interceptors
import { AuthProvider } from "./auth/AuthContext";
// … render <AuthProvider><App /></AuthProvider>
Why the single-flight queue? If a page fires 5 requests at once and all get 401, you don't want 5 parallel /refresh calls — the first rotates the cookie and the other four would fail with a now-revoked token. The queue makes exactly one refresh happen; the other four wait for its result and then replay.

9. React — restoring the session on reload

Because the access token lives only in memory, a page refresh wipes it. But the refresh cookie survives. So on app boot we make a single /refresh call (already wired in the AuthProvider useEffect above):

  • Cookie still valid → we get a fresh access token, restore admin, user stays logged in.
  • Cookie missing/expired → /refresh returns 401, we stay logged out and show the login screen.

Render a splash/loader while booting is true so the app doesn't flash the login page before the boot refresh resolves:

function App() {
  const { booting } = useAuth();
  if (booting) return <FullScreenLoader />;   // avoids a login-page flash on reload
  return <Routes>{/* … */}</Routes>;
}

10. React — logout

const { logout } = useAuth();
// …
<button onClick={async () => { await logout(); navigate("/login"); }}>Log out</button>

The /admin/logout call revokes the current access token and the paired refresh token on the server, and the response clears the cookie. Locally we also drop the in-memory access token and the admin state.

11. React — protected routes

src/auth/RequireAuth.jsx 
import { Navigate, useLocation } from "react-router-dom";
import { useAuth } from "./AuthContext";

export function RequireAuth({ children }) {
  const { isAuthenticated, booting } = useAuth();
  const location = useLocation();

  if (booting) return <FullScreenLoader />;
  if (!isAuthenticated) return <Navigate to="/login" state={{ from: location }} replace />;
  return children;
}

12. What happens when the access token expires

This is the exact step-by-step the interceptor performs — the user sees none of it:

 1.  Component calls   api.get("/admin/profile")
 2.  Request interceptor attaches  Authorization: Bearer <expired access>
 3.  API returns         401 Unauthorized        (token past its 10-min expiry)
 4.  Response interceptor sees 401, not yet retried, not the /refresh call
 5.  No refresh in flight -> set isRefreshing = true
 6.  Interceptor calls   POST /admin/refresh
        • browser auto-attaches  Cookie: refresh_token=<valid>
 7.  API validates the refresh token, checks ability:refresh, not expired
 8.  API ROTATES: deletes the old refresh token, issues:
        • new access  token (10 min)  -> JSON body
        • new refresh token (60 min)  -> Set-Cookie (replaces the old cookie)
 9.  Interceptor stores new access token in memory, updates React state
10.  Interceptor replays step 1 with  Authorization: Bearer <new access>
11.  API returns         200 OK  + the profile data
12.  The original api.get(...) promise resolves as if nothing happened

If step 6 returns 401 instead (refresh token expired after 60 min idle, or it was revoked), the interceptor clears local auth state and the route guard redirects to /login.

13. Dual-mode: web vs native apps

An httpOnly cookie is a browser mechanism — a native Android/iOS app has no cookie jar and no document.cookie, so the cookie transport doesn't apply. The same API therefore runs in two modes. Everything server-side is identical (same tokens, same refresh/access abilities, same single-use rotation, same sliding 60-minute window) — only the carrier of the refresh token changes.

 Web (browser)Native app (Android / iOS)
Login header—X-Client: mobile
Refresh token delivered inSet-Cookie (httpOnly)JSON body data.refresh_token
App stores it inbrowser cookie jar (automatic)Keychain (iOS) / Keystore (Android)
Sent on refresh / logout viacookie (automatic)X-Refresh-Token header
Refresh responsenew Set-Cookienew data.refresh_token
Access tokenidentical — JSON body → in-memory variable, sent as Authorization: Bearer
How the server picks the mode. On login the client opts in with the X-Client: mobile header. On refresh the server simply mirrors the transport it received: a refresh token arriving in a cookie gets a cookie back; one arriving in the X-Refresh-Token header (or request body) gets the new token back in the JSON body. So a mobile client doesn't even need X-Client on refresh — using the header is enough.

Mobile request/response shapes

// Login — mobile opts in via header
POST /api/admin/login
X-Client: mobile
{ "email": "…", "password": "…", "device_name": "Pixel 8 / iOS 18" }

// 200 — BOTH tokens in the body, no Set-Cookie
{ "data": { "access_token": "…", "refresh_token": "…", "admin": { … } } }

// Refresh — present the stored refresh token in a header
POST /api/admin/refresh
X-Refresh-Token: <refresh token from Keychain/Keystore>

// 200 — NEW pair in the body (old refresh token is now revoked)
{ "data": { "access_token": "…(new)", "refresh_token": "…(new)", "admin": { … } } }

// Logout — send both so the server revokes the pair
POST /api/admin/logout
Authorization: Bearer <access token>
X-Refresh-Token: <refresh token>
The #1 mobile bug: rotation means every successful /refresh returns a new refresh token and revokes the old one. The app must persist the new token to secure storage immediately, overwriting the old value. Miss this and the next refresh fails with 401.

14. Mobile — where to store the token

The access token is just an in-memory variable (lost on app kill — you re-acquire it on next launch via refresh). The refresh token must go in platform secure storage, never in plain SharedPreferences, UserDefaults, or a file.

PlatformStore the refresh token inBacked by
Android (native)EncryptedSharedPreferences (Jetpack Security) or DataStoreAndroid Keystore (hardware-backed)
iOS (native)Keychain — kSecClassGenericPassword, accessibility WhenUnlockedThisDeviceOnlySecure Enclave / Keychain
Flutterflutter_secure_storageKeystore / Keychain
React Nativereact-native-keychain or Expo SecureStoreKeystore / Keychain
On mobile, httpOnly gives you nothing (there's no JS to defend against), so the protection comes entirely from secure storage + short access-token lifetime + rotation. Storing the refresh token in plain prefs/files is the mobile equivalent of putting a web token in localStorage.

15. Mobile — login & refresh (Kotlin / Swift)

The logic mirrors the web interceptor: attach the access token, catch 401, refresh once (single-flight), overwrite the stored tokens, retry.

Android — OkHttp Authenticator (auto-fires on 401)

TokenAuthenticator.kt 
class TokenAuthenticator(
    private val store: SecureTokenStore,   // wraps EncryptedSharedPreferences / Keystore
    private val api: AuthApi,
) : Authenticator {

    override fun authenticate(route: Route?, response: Response): Request? {
        synchronized(this) {                       // single-flight: one refresh at a time
            val current = store.accessToken
            // Another thread may have already refreshed while we waited on the lock.
            if (response.request.header("Authorization") != "Bearer $current") {
                return response.request.newBuilder()
                    .header("Authorization", "Bearer $current").build()
            }
            val refreshed = try {
                api.refresh(store.refreshToken)         // sends X-Refresh-Token
            } catch (e: Exception) { null } ?: run {
                store.clear(); return null             // refresh failed -> give up -> route to login
            }
            store.accessToken  = refreshed.accessToken
            store.refreshToken = refreshed.refreshToken // ROTATION: overwrite immediately
            return response.request.newBuilder()
                .header("Authorization", "Bearer ${refreshed.accessToken}").build()
        }
    }
}

// Retrofit interface — note the header carries the stored refresh token.
interface AuthApi {
    @Headers("X-Client: mobile")
    @POST("admin/login")
    suspend fun login(@Body body: LoginBody): TokenPair

    @POST("admin/refresh")
    fun refresh(@Header("X-Refresh-Token") refreshToken: String): TokenPair
}

Wire it up with OkHttpClient.Builder().authenticator(tokenAuthenticator) plus a request interceptor that adds Authorization: Bearer <access> to every call.

iOS — URLSession retry wrapper

AuthClient.swift 
func authedRequest(_ build: () -> URLRequest) async throws -> Data {
    var req = build()
    req.setValue("Bearer \(store.accessToken)", forHTTPHeaderField: "Authorization")
    var (data, resp) = try await URLSession.shared.data(for: req)

    if (resp as? HTTPURLResponse)?.statusCode == 401 {
        try await refreshOnce()                         // actor-guarded single-flight
        req.setValue("Bearer \(store.accessToken)", forHTTPHeaderField: "Authorization")
        (data, resp) = try await URLSession.shared.data(for: req)  // retry once
    }
    return data
}

func refreshOnce() async throws {
    var r = URLRequest(url: api("/admin/refresh"))
    r.httpMethod = "POST"
    r.setValue(store.refreshToken, forHTTPHeaderField: "X-Refresh-Token")   // from Keychain
    let (data, resp) = try await URLSession.shared.data(for: r)
    guard (resp as? HTTPURLResponse)?.statusCode == 200 else {
        store.clear(); throw AuthError.sessionExpired        // -> show login
    }
    let pair = try JSONDecoder().decode(TokenEnvelope.self, from: data).data
    store.accessToken  = pair.accessToken
    store.refreshToken = pair.refreshToken                      // ROTATION: overwrite in Keychain
}
Tip — longer lifetime for mobile. A 60-minute refresh window means a phone forces re-login after an hour idle, which is poor UX. Apps commonly use multi-day/30-day refresh tokens. You can vary the expiry by client type inside AdminAuthService::issueTokenPair() (e.g. read the X-Client header and pick a longer TTL) without touching anything else.

16. Security notes

ThreatMitigation in this design
XSS stealing a long-lived tokenRefresh token is httpOnly — unreadable by JS. Access token is short-lived (10 min) and memory-only.
Refresh token replay / theftSingle-use rotation — each /refresh revokes the presented token. A stolen-and-replayed token is already dead.
CSRF on the cookie-based refreshSameSite=Strict + cookie path scoped to /api/admin. The cookie isn't sent on cross-site requests.
Refresh token used to call business APIsability:access middleware — the refresh token only holds the refresh ability, so it's rejected (403) everywhere except /refresh.
Token interception over the wireSecure flag — cookie only sent over HTTPS in production.
Brute-forcing the refresh endpointthrottle:admin-refresh (20/min per IP).
Native app token theft (no httpOnly)Refresh token in OS secure storage (Keychain / Keystore), short-lived access token, single-use rotation — the same protections, minus the browser-only cookie.
Never put either token in localStorage or sessionStorage, and never return the refresh token in a JSON body. Both undo the XSS protection that the whole design exists to provide.

17. Environment & production

Backend .env

SANCTUM_ACCESS_TOKEN_EXPIRATION=10     # minutes (default 10)
SANCTUM_REFRESH_TOKEN_EXPIRATION=60    # minutes (default 60)

Prune expired tokens

Schedule Sanctum's pruning command so revoked/expired rows don't pile up (routes/console.php):

use Illuminate\Support\Facades\Schedule;
Schedule::command('sanctum:prune-expired --hours=24')->daily();

Same-site vs cross-site cookies

  • Same registrable domain (e.g. admin.example.com ↔ api.example.com, or localhost:3000 ↔ localhost:8000) → SameSite=Strict works as-is. Ports don't affect same-site.
  • ✕ Truly different domains → the Strict cookie won't be sent. You'd switch the cookie to SameSite=None; Secure and add explicit CSRF protection on /refresh and /logout.

18. FAQ / troubleshooting

The refresh cookie is never sent — every /refresh returns 401.

You forgot withCredentials: true on the axios instance, or the API's CORS config doesn't have supports_credentials => true with your exact SPA origin listed.

Login works but the user is logged out on every page reload.

The boot /refresh call isn't firing or the cookie isn't being stored. Check the Network tab: the login response should include a Set-Cookie: refresh_token=…; HttpOnly header, and the boot /refresh request should carry it back.

I can't see the refresh token in document.cookie.

Correct — that's the point. httpOnly cookies are invisible to JavaScript. Inspect it under DevTools → Application → Cookies instead.

After logging in, calls work for 10 minutes then a request hangs/duplicates.

Make sure the interceptor's single-flight queue is in place (section 8). Parallel 401s without it trigger multiple refreshes, and the rotation makes all but the first fail.

Why is device_name optional?

It's a human-readable label for the token (e.g. "Chrome on macOS"), useful if you later build a "signed-in devices" screen. Omit it and it defaults to admin.

(Mobile) The app logs out after the first refresh.

You're not persisting the rotated token. Each /refresh returns a new refresh_token and revokes the old one — the app must overwrite the stored value in Keychain/Keystore right away. Reusing the old token gives a 401. See §13.

(Mobile) Login returns no refresh_token in the body.

You didn't send the X-Client: mobile header, so the server used browser mode and issued a cookie instead. Add the header to the login request.
