Auth Approach Comparison

An evaluation of two React + Laravel-Sanctum admin starter kits — the existing DebugPeople/react-starter-kit vs. the planned starter_kit_frontend design — against the project's hard requirements and the current starter_kit_api backend. Verdict and a "best of both" recommendation included.

1. Executive summary

Both kits target the same backend family and share a modern stack (React 19, TanStack Query, Zustand, Sonner, axios). They diverge on the one thing that matters most for an admin panel: how the auth token is stored and refreshed.

  • The DebugPeople kit stores the access token in localStorage and has no refresh-token flow — a single long-lived token.
  • The starter_kit_frontend design keeps the access token in memory only (Zustand) and relies on an httpOnly refresh cookie + silent-refresh interceptor.

Bottom line

For this project's stated requirement ("do not store the access token in localStorage") and the current starter_kit_api (short-lived access token + httpOnly refresh cookie), the starter_kit_frontend design is the correct approach. The DebugPeople kit is disqualified on two counts: it violates the no-localStorage rule, and it was built against an older API that returned a single token — it would not even authenticate against the current backend. Adopt the starter_kit_frontend auth architecture, and borrow the DebugPeople kit's class-based service layer and Zod validation.

4. The decisive issue — localStorage

The DebugPeople kit persists the bearer token (and the full profile) to localStorage, and the axios interceptor reads it from there:

DebugPeople · src/store/useAuthStore.ts
setUser: (user, token) => {
  localStorage.setItem('token', token);          // ← the access token, in web storage
  localStorage.setItem('profile', JSON.stringify(user));
  set({ user, isAuthenticated: true });
},
DebugPeople · src/services/base.ts
this.http.interceptors.request.use((config) => {
  const token = localStorage.getItem('token');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});
This is the project's explicit no-go. The requirement was "do not store the access token in localStorage." Beyond the rule, it is a real security weakness:
  • XSS exposure: anything in localStorage is readable by any injected script — token theft is trivial. An in-memory token is not script-reachable from storage, and the refresh token is httpOnly (unreadable by JS at all).
  • No rotation: a single long-lived token can't be short-lived without logging the user out, so it tends to be long-lived — widening the theft window.

The starter_kit_frontend design holds the token only in the Zustand store's memory and reads it via getState() in the interceptor — nothing reaches storage.

5. Backend compatibility

This is the point that settles it on engineering grounds, independent of preference. The current starter_kit_api AuthController returns:

data: { "access_token": "...", "admin": { ... } }   // refresh token delivered as an httpOnly cookie

The DebugPeople kit expects the older shape and reads a field that no longer exists:

DebugPeople · src/store/useAuthStore.ts (login)
onSuccess: ({ data: { token, admin } }) => {   // ← `token` is undefined on the current API
  setUser(admin, token);
}
The DebugPeople kit predates the refresh-token feature. The backend's most recent commit is literally "refresh token is implemented", switching the payload to access_token + a short (~10 min) access TTL and a ~60 min httpOnly refresh cookie. As written, the DebugPeople kit would receive undefined for the token and fail to authenticate against the current backend. The starter_kit_frontend design is written against the backend as it exists today.

6. Single source of truth

A subtler architectural difference: in the DebugPeople kit, the Zustand store is not really the source of truth — localStorage is, and the store is initialized from it on every page load:

DebugPeople · src/store/useAuthStore.ts
export const useAuthStore = create<AuthState>((set) => ({
  user: loadUser(),                                  // parsed from localStorage
  isAuthenticated: !!localStorage.getItem('token'),  // derived from localStorage
  ...
}));

State now lives in two places that must be kept in sync by hand on every write. The starter_kit_frontend design avoids this entirely: the store holds the session in memory as the one authority, and the httpOnly cookie (owned by the server) restores it on reload via bootstrap() — no duplicated client state.

7. Where each approach wins

starter_kit_frontend wins on

  • Token security (in-memory + httpOnly refresh)
  • Seamless UX on short token TTL (silent refresh)
  • Single source of truth for the session
  • Correctness against the current backend
  • Compliance with the project requirement

DebugPeople kit wins on

  • Class-based BaseService — DRY, typed transport
  • Zod schemas for validation
  • End-to-end TypeScript
  • Clean per-resource hook co-location
  • Tidy envelope helpers (withMessage variants)

8. What to borrow from the DebugPeople kit

8.1 Class-based BaseService transport

Their typed wrapper around the { data, message, success } envelope is cleaner than repeating .then(r => r.data.data) in every feature file. Adopt it as the transport, with one shared axios instance (their version news an instance per service — consolidate to one):

borrowed pattern · lib/base-service.js (single shared `api`)
import { api } from "@/lib/api";

export class BaseService {
  get(url, config)            { return api.get(url, config).then((r) => r.data.data); }
  post(url, body, config)     { return api.post(url, body, config).then((r) => r.data.data); }
  postWithMessage(url, body)  { return api.post(url, body).then((r) => ({ data: r.data.data, message: r.data.message })); }
  put(url, body)              { return api.put(url, body).then((r) => r.data.data); }
  del(url)                    { return api.delete(url).then((r) => r.data.data); }
}

Keep the silent-refresh interceptor on the shared api instance — so the borrowed transport still benefits from in-memory tokens + refresh.

8.2 Zod validation via the react-hook-form resolver

Take Zod (stronger, reusable schemas) but pair it with react-hook-form instead of the custom useFormValidation hook — RHF is more capable and is already the documented form layer:

npm install zod @hookform/resolvers
import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import { z } from "zod";

const loginSchema = z.object({
  email: z.string().email(),
  password: z.string().min(1),
});

const { register, handleSubmit, formState } = useForm({
  resolver: zodResolver(loginSchema),
});

8.3 Per-resource hook co-location

Both designs already do this — their store/useAdminUserStore.ts (a bundle of useQuery/useMutation + form hooks) is the same idea as the planned features/admin-users/*.hooks.js. No change needed; just confirms the structure.

9. Verdict & recommendation

Recommendation

Build on the starter_kit_frontend auth architecture — in-memory access token, httpOnly refresh cookie, silent-refresh interceptor, Zustand as the single session source, TanStack Query for server data. It is the only approach that satisfies the no-localStorage requirement and works with the current backend.

Graft on two patterns from the DebugPeople kit: (1) a class-based BaseService on top of the shared, interceptor-equipped axios instance, and (2) Zod validation through the react-hook-form resolver. The result keeps the secure token model while gaining a DRY transport layer and schema-based validation.
