1. Goals & constraints

This document is the implementation plan for authentication in starter_kit_frontend. The rules driving every design choice:

  • Access token lives in memory only. Never localStorage / sessionStorage. It is held in the Zustand auth store; the axios interceptor reads it via useAuthStore.getState().token — synchronously, outside React, with no separate module variable to keep in sync.
  • Refresh token is never touched by JS. The API already delivers it as an httpOnly cookie scoped to /api/admin for browser clients — the SPA cannot and should not read it.
  • State is served by Zustand stores. useAuthStore holds user, token, status and the auth actions; useThemeStore holds theme + toggle. Stores are module singletons — no Context, no providers.
  • Axios interceptors attach the bearer token on the way out and perform a single silent refresh on a 401 on the way back.
  • TanStack Query owns all cached server data (lists, details, writes). Zustand owns client/session state (auth, theme). React Hook Form owns form state.
  • Toaster for user feedback (login errors, success, session expiry) using sonner.
Why in-memory is safe here: because we keep the access token out of storage, an XSS-injected script can't read it from localStorage, and the refresh token is httpOnly so script can't read that either. The trade-off — the access token is lost on full page reload — is solved by a silent refresh on app boot (the cookie re-mints a fresh access token). See §4.
Why Zustand instead of Context. The token must be read synchronously and outside React by the axios interceptor. With Context you'd need a second module-level variable mirrored to React state. A Zustand store is one object that is both reactive inside components (useAuthStore(selector)) and readable imperatively outside React (useAuthStore.getState()) — so it is the single source of truth for both worlds and removes the dual-write. It also needs no provider wrapper.

2. API analysis (starter_kit_api)

All admin auth routes are defined in routes/api.php under the admin prefix. With VITE_SERVER_BASE_URL = http://localhost:8080/api, the SPA calls /admin/<route>.

Method & pathAuthRequest bodyPurpose
POST /admin/loginpublicemail, password, device_name?Authenticate; returns access token + admin, sets refresh cookie.
POST /admin/refreshcookie only(none — cookie carries it)Mint a new access token from the refresh cookie. Rotates the cookie.
POST /admin/logoutbearer(none)Revoke the session; clears refresh cookie.
POST /admin/forgot-passwordpublicemailEmail a 6-digit OTP (generic response, no enumeration).
POST /admin/reset-passwordpublicemail, otp, password, password_confirmationVerify OTP & set a new password.
POST /admin/change-passwordbearercurrent_password, password, password_confirmationChange password; signs out other sessions.
GET /admin/profilebearer—Current admin + roles + flattened permissions.
PUT /admin/profilebearername, emailUpdate own profile.
Key behaviours observed in AuthController:
  • The refresh token is returned in the JSON body only when the client sends header X-Client: mobile. A browser omits it, so the token comes back as an httpOnly; SameSite=strict cookie at path /api/admin. We are a browser client — do not send X-Client.
  • /admin/refresh reads the cookie automatically, so axios must send credentials (withCredentials: true).
  • Access token TTL is short (SANCTUM_ACCESS_TOKEN_EXPIRATION, default 10 min); refresh TTL default 60 min. Expect frequent 401s — the interceptor must handle them transparently.
  • Refresh tokens carry only the refresh ability; access tokens carry access. Business endpoints sit behind ability:access, so a refresh token can only hit /refresh.

3. Response envelope & payloads

Every response uses the ApiResponder trait. Success and error shapes:

success — login / refresh
{
  "success": true,
  "message": "Login successful.",
  "data": {
    "access_token": "12|abcd...",
    "admin": {
      "id": 1,
      "name": "Admin",
      "email": "admin@example.com",
      "user_type": "admin",
      "user_type_label": "Admin",
      "status": "active",
      "status_label": "Active",
      "roles": [ { "name": "super-admin", "status_label": "Active" } ],
      "permissions": [ "roles.view", "admins.view" ]
    }
    // NOTE: no refresh_token for browser clients
  }
}
error — validation / auth failure
{
  "success": false,
  "message": "Invalid credentials.",
  "errors": {
    "email": ["The email field is required."]
  }
}

Login bad-credentials → 422; deactivated → 403; expired/invalid refresh → 401 (and the cookie is cleared server-side).

Two different data shapes — watch this. Login/refresh nest the user under data.admin. But GET /admin/profile returns the AdminResource directly as data (no admin wrapper). Our API helpers must read each correctly.

4. Token storage strategy

zustand Access token

Held in the useAuthStore state (memory). Components read it reactively; the axios interceptor reads it via useAuthStore.getState().token. Not persisted — lost on reload, re-minted by bootstrap.

httpOnly cookie Refresh token

Set & rotated by the server, scoped to /api/admin. JS never sees it. Survives reload, so it bootstraps a new session silently.

Theme is the one thing we persist. useThemeStore uses Zustand's persist middleware to keep the choice in localStorage (key sk-theme) — theme is non-sensitive, unlike the token which is deliberately never persisted.

8. Axios instance + interceptors

8.1 Token source: the auth store

There is no separate token module — the interceptor reads the token straight from the Zustand store with getState(), which works outside React. On a failed refresh it calls the store's clear() directly, so there's no callback-registration plumbing.

Import cycle is safe. api.js imports useAuthStore; the store imports auth.api.js, which imports api.js. ES modules handle this because every cross-reference is used inside a function at runtime (the interceptor callback, the store actions), never at module-evaluation time.

8.2 The axios instance

src/lib/api.js
import axios from "axios";
import { useAuthStore } from "@/features/auth/auth.store";

const BASE = import.meta.env.VITE_SERVER_BASE_URL;

export const api = axios.create({
  baseURL: BASE,
  withCredentials: true,            // send/receive the httpOnly refresh cookie
  headers: { Accept: "application/json" },
});

// ---- Request: attach the in-memory bearer token (read straight from the store) ----
api.interceptors.request.use((config) => {
  const token = useAuthStore.getState().token;   // synchronous, outside React
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// ---- Response: silent single-flight refresh on 401 ----
let isRefreshing = false;
let waiters = [];                    // requests parked while a refresh is in flight
const flushWaiters = (token) => { waiters.forEach((w) => w(token)); waiters = []; };

api.interceptors.response.use(
  (res) => res,
  async (error) => {
    const { response, config } = error;
    const original = config || {};

    // Only handle 401s that aren't the refresh call itself and haven't retried.
    const isRefreshCall = original.url?.includes("/admin/refresh");
    if (response?.status !== 401 || original._retried || isRefreshCall) {
      return Promise.reject(error);
    }
    original._retried = true;

    // A refresh is already running — wait for its result, then replay.
    if (isRefreshing) {
      return new Promise((resolve, reject) => {
        waiters.push((token) => {
          if (!token) return reject(error);
          original.headers.Authorization = `Bearer ${token}`;
          resolve(api(original));
        });
      });
    }

    isRefreshing = true;
    try {
      // Bare axios (not `api`) to skip interceptors; cookie carries the refresh token.
      const { data } = await axios.post(`${BASE}/admin/refresh`, {}, {
        withCredentials: true,
        headers: { Accept: "application/json" },
      });
      const newToken = data?.data?.access_token ?? null;
      useAuthStore.getState().setToken(newToken);
      flushWaiters(newToken);
      original.headers.Authorization = `Bearer ${newToken}`;
      return api(original);
    } catch (refreshError) {
      useAuthStore.getState().clear();   // reset session + toast; guard redirects to /login
      flushWaiters(null);
      return Promise.reject(refreshError);
    } finally {
      isRefreshing = false;
    }
  },
);
Single-flight matters: if a dashboard fires 5 requests and all 401 at once, only the first triggers /refresh; the other four park in waiters and replay with the new token. Without this you'd hammer /refresh and possibly rotate the cookie into a race.

8.3 BaseService — the shared transport class

Instead of repeating .then((r) => r.data.data) in every feature, a small BaseService unwraps the { success, message, data } envelope once. Because the envelope always carries a message, there's no need for a separate *WithMessage variant — every method returns the whole envelope, and callers destructure { data } or { data, message } as needed. It runs on the same shared api instance from §8.2, so it inherits the bearer token + silent-refresh interceptor (unlike the borrowed original, which created one axios instance per service).

src/lib/base-service.js
import { api } from "@/lib/api";

// Every method resolves to the API envelope: { success, message, data }.
// Read `.data` for the payload; read `.message` when you want to toast it.
export class BaseService {
  get(url, config)        { return api.get(url, config).then((r) => r.data); }
  post(url, body, config) { return api.post(url, body, config).then((r) => r.data); }
  put(url, body, config)  { return api.put(url, body, config).then((r) => r.data); }
  del(url, config)        { return api.delete(url, config).then((r) => r.data); }
}
Why one shape, not a conditional. A method that sometimes returns data and sometimes { data, message } forces callers to know which — fragile. Returning the same envelope everywhere is predictable: the payload is always res.data, the message always res.message.

8.4 Auth service (extends BaseService)

Auth's transport lives at features/auth/auth.api.js as a class extending BaseService, exported as a ready instance. Each method resolves to the envelope; callers read .data (and .message where useful). These feed the auth store (§10); the password-flow methods can also be wrapped in useMutation from auth.hooks.js.

src/features/auth/auth.api.js
import { BaseService } from "@/lib/base-service";

class AuthService extends BaseService {
  login(email, password) { return this.post("/admin/login", { email, password }); }   // data: { access_token, admin }
  refresh()              { return this.post("/admin/refresh"); }                       // data: { access_token, admin }
  logout()               { return this.post("/admin/logout"); }

  profile()              { return this.get("/admin/profile"); }                         // data: AdminResource
  updateProfile(payload) { return this.put("/admin/profile", payload); }

  forgotPassword(email)  { return this.post("/admin/forgot-password", { email }); }
  resetPassword(payload) { return this.post("/admin/reset-password", payload); }   // { email, otp, password, password_confirmation }
  changePassword(payload){ return this.post("/admin/change-password", payload); }  // { current_password, password, password_confirmation }
}

export const authApi = new AuthService();

9. Data layer — TanStack Query

All server state — lists, detail records, and writes — goes through TanStack Query. The axios instance from §8 stays as the transport layer; TanStack Query simply calls into it and adds caching, background refresh, loading/error state, and cache invalidation. @tanstack/react-query is already installed; add the devtools too.

9.1 Who owns what: Query vs Zustand

"TanStack Query for all API calls" does not mean it owns auth identity. The split:

StateOwnerExamples
Server state (cached, refetchable)TanStack Query (useQuery)admin-users, roles, todos, audit logs, profile view
WritesTanStack Query (useMutation)create / edit / delete, change password
Session / identityZustand (useAuthStore)user, in-memory token, status
UI / preferenceZustand (useThemeStore) / local statetheme, modals, form state (RHF)
Why auth lives in Zustand, not the query cache. Route guards and the axios interceptor need the token / status synchronously — the interceptor reads it from getState() outside any React render. That's identity state, not cached server data. Login/logout/refresh are imperative session transitions (set token + user + status together), which is exactly a store action — not a passive cache entry. So auth has no query key; its results land in the store.

9.4 Recommended structure — one hook per operation

Don't scatter inline useQuery({ queryKey, queryFn }) across components. Wrap each operation in a named hook, with a query-key factory per resource so invalidation never breaks on a typo. Co-locate per feature:

src/features/admin-users/
├─ admin-users.api.js     # raw axios calls (transport)
├─ admin-users.keys.js    # query-key factory
└─ admin-users.hooks.js   # useAdminUsers(), useAdminUser(id), useSaveAdminUser(id)
admin-users.keys.js
export const adminUserKeys = {
  all: ["admin-users"],
  list: (filters) => ["admin-users", "list", filters],
  detail: (id) => ["admin-users", "detail", id],
};
admin-users.api.js (extends BaseService)
import { BaseService } from "@/lib/base-service";

class AdminUserService extends BaseService {
  list(filters)        { return this.get("/admin/admin-users", { params: filters }); }
  getOne(id)           { return this.get(`/admin/admin-users/${id}`); }
  create(body)         { return this.post("/admin/admin-users", body); }
  update(id, body)     { return this.put(`/admin/admin-users/${id}`, body); }
  remove(id)           { return this.del(`/admin/admin-users/${id}`); }
}

export const adminUserApi = new AdminUserService();
admin-users.hooks.js
import { useQuery, useMutation, useQueryClient } from "@tanstack/react-query";
import { toast } from "sonner";
import { adminUserKeys } from "./admin-users.keys";
import { adminUserApi } from "./admin-users.api";

export function useAdminUsers(filters) {
  return useQuery({
    queryKey: adminUserKeys.list(filters),
    queryFn: () => adminUserApi.list(filters).then((r) => r.data),   // unwrap envelope
  });
}

export function useAdminUser(id) {
  return useQuery({
    queryKey: adminUserKeys.detail(id),
    queryFn: () => adminUserApi.getOne(id).then((r) => r.data),
    enabled: Boolean(id),            // skip when creating
  });
}

export function useSaveAdminUser(id) {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (values) => (id ? adminUserApi.update(id, values) : adminUserApi.create(values)),
    onSuccess: (res) => {            // res = { success, message, data }
      qc.invalidateQueries({ queryKey: adminUserKeys.all });   // refresh the list
      toast.success(res.message);    // the API message — no separate WithMessage call needed
    },
  });
}

Components stay clean — they never touch axios or query keys:

const { data, isLoading } = useAdminUsers(filters);     // list page
const { mutateAsync, isPending } = useSaveAdminUser(id); // form page

9.5 Forms: react-hook-form + Zod + useMutation (+ useQuery to prefill)

Validation rules live once as a Zod schema in validation/, fed to react-hook-form via zodResolver. The submit is a useMutation; editing adds a useQuery to load and reset() to prefill. One component handles both create and edit.

src/validation/admin-users.js
import { z } from "zod";

export const adminUserCreateSchema = z.object({
  name: z.string().min(1, "Name is required"),
  email: z.string().email("Enter a valid email"),
  role: z.string().min(1, "Pick a role"),
});

// edit reuses the same shape (extend/partial as needed)
export const adminUserUpdateSchema = adminUserCreateSchema;
src/pages/admins/AdminUserForm.jsx (core)
import { useEffect } from "react";
import { useForm } from "react-hook-form";
import { zodResolver } from "@hookform/resolvers/zod";
import { useNavigate } from "react-router";
import { toast } from "sonner";
import { useAdminUser, useSaveAdminUser } from "@/features/admin-users/admin-users.hooks";
import { adminUserCreateSchema } from "@/validation/admin-users";
import { apiError } from "@/lib/errors";
import { ROUTES } from "@/routes/path";

export default function AdminUserForm({ adminId }) {     // id present = edit
  const navigate = useNavigate();
  const isEdit = Boolean(adminId);

  const { data: existing, isLoading } = useAdminUser(adminId);   // auto-skipped on create
  const { register, handleSubmit, reset, setError, formState: { errors } } = useForm({
    resolver: zodResolver(adminUserCreateSchema),    // Zod validates the form
    defaultValues: { name: "", email: "", role: "" },
  });

  useEffect(() => { if (existing) reset(existing); }, [existing, reset]);   // prefill

  const { mutateAsync, isPending } = useSaveAdminUser(adminId);

  const onSubmit = async (values) => {   // only runs once Zod passes
    try {
      await mutateAsync(values);       // success toast (API message) + list refresh handled in the hook
      navigate(ROUTES.admins?.index ?? "/admin-users");
    } catch (err) {
      // Map Laravel 422 field errors back onto the form…
      const fieldErrors = err?.response?.data?.errors;
      if (fieldErrors) {
        Object.entries(fieldErrors).forEach(([field, msgs]) => setError(field, { message: msgs[0] }));
      } else {
        toast.error(apiError(err));   // …otherwise toast the message
      }
    }
  };

  if (isEdit && isLoading) return <div>Loading…</div>;

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register("name")} />
      {errors.name && <span>{errors.name.message}</span>}   // message comes from the Zod schema
      <input {...register("email")} />
      {errors.email && <span>{errors.email.message}</span>}
      <button disabled={isPending}>{isPending ? "Saving…" : isEdit ? "Update" : "Create"}</button>
    </form>
  );
}
Why Zod-via-resolver: the rules (and messages) live in one reusable schema instead of being scattered inside each register(...) call. handleSubmit runs the schema first, so onSubmit only fires with valid data, and errors.<field>.message is populated straight from the schema. (Server-side 422s are still mapped back via setError.)
When you can skip useMutation: only when the write has no cached query to refresh — e.g. the pure auth forms (login, forgot/reset password) whose results go into the auth store, not the query cache. Those still use react-hook-form + zodResolver, calling authApi (or a thin useMutation in auth.hooks.js).

9.6 Freshness knobs (per query)

GoalOptions
Reuse cache within a windowstaleTime: 5 * 60_000
Refetch on every visitrefetchOnMount: "always" (or staleTime: 0)
Auto-refresh while the page sits idlerefetchInterval: 5 * 60_000 (active timer — staleTime alone won't do this)
Keep / evict cache between visitsgcTime (default 5 min)
Multi-admin: see other admins' editspoll with refetchInterval, or push via WebSockets then invalidateQueries
Cache is per-browser, client-side. invalidateQueries only refreshes the current user's cache — it cannot reach another admin's browser. For concurrent-edit freshness, use refetchInterval (polling) on the critical lists, or Laravel Reverb/Echo for real-time push.

10. Auth store (Zustand)

The store is a module singleton — no provider, no Context. It holds session state and the async auth actions; the axios interceptor (§8) reads and clears it via getState().

src/features/auth/auth.store.js
import { create } from "zustand";
import { toast } from "sonner";
import { authApi } from "./auth.api";

// Guards the boot refresh against React 18/19 StrictMode's double-invoked effect.
let bootstrapStarted = false;

// status: "loading" (bootstrapping) | "authenticated" | "unauthenticated"
export const useAuthStore = create((set, get) => ({
  user: null,
  token: null,                       // in memory only — NEVER persisted
  status: "loading",

  setToken: (token) => set({ token }),
  setUser: (user) => set({ user }),

  isAuthenticated: () => get().status === "authenticated",

  // Called once on app load: mint an access token from the refresh cookie.
  bootstrap: async () => {
    if (bootstrapStarted) return;
    bootstrapStarted = true;
    try {
      const { data } = await authApi.refresh();       // envelope → data: { access_token, admin }
      set({ token: data.access_token, user: data.admin, status: "authenticated" });
    } catch {
      set({ token: null, user: null, status: "unauthenticated" });
    }
  },

  login: async (email, password) => {
    const { data } = await authApi.login(email, password);  // throws on 4xx → caller toasts
    set({ token: data.access_token, user: data.admin, status: "authenticated" });
    return data.admin;
  },

  logout: async () => {
    try { await authApi.logout(); } catch { /* best-effort */ }
    set({ token: null, user: null, status: "unauthenticated" });
  },

  // Called by the axios interceptor when a refresh ultimately fails.
  clear: () => {
    set({ token: null, user: null, status: "unauthenticated" });
    toast.error("Your session has expired. Please sign in again.");
  },
}));
Do NOT add the persist middleware to this store. The whole point is that the token never reaches storage. The session is re-established on reload by bootstrap() via the httpOnly cookie. (Only the theme store persists — §12.)
Reading the store. Inside components, select the slice you need: const user = useAuthStore((s) => s.user). Outside React (interceptor): useAuthStore.getState().token. Calling actions anywhere: useAuthStore.getState().logout().

11. Store hooks (useAuth / useTheme)

Thin selector hooks give screens and guards an ergonomic, stable shape and keep the store's exact key names in one place. useShallow prevents needless re-renders when selecting an object of several fields.

      login: s.login,
      logout: s.logout,
      setUser: s.setUser,
    })),
  );
}

import { useShallow } from "zustand/react/shallow";
import { useThemeStore } from "@/store/theme.store";

export function useTheme() {

12. Theme store (Zustand)

Theme is not sensitive, so it uses the persist middleware to remember the choice in localStorage. The store also applies the dark class to <html> whenever the theme changes and on rehydration.

src/store/theme.store.js
import { create } from "zustand";
import { persist } from "zustand/middleware";

const applyTheme = (theme) =>
  document.documentElement.classList.toggle("dark", theme === "dark");

const systemPrefersDark = () =>
  window.matchMedia("(prefers-color-scheme: dark)").matches;

export const useThemeStore = create(
  persist(
    (set, get) => ({
      theme: systemPrefersDark() ? "dark" : "light",
      setTheme: (theme) => { applyTheme(theme); set({ theme }); },
      toggle: () => {
        const next = get().theme === "dark" ? "light" : "dark";
        applyTheme(next);
        set({ theme: next });
      },
    }),
    {
      name: "sk-theme",                       // localStorage key
      onRehydrateStorage: () => (state) => {  // apply the saved theme on load
        if (state) applyTheme(state.theme);
      },
    },
  ),
);
Because persist hydrates synchronously on first import, the dark class is applied as soon as the store is first used (the Toaster in App.jsx reads useTheme, so it imports early). No ThemeProvider and no boot effect are needed.

src/lib/errors.js
// Pull the API's `message`, else the first validation error, else a fallback.
export function apiError(error, fallback = "Something went wrong.") {
  const data = error?.response?.data;
  if (data?.message) return data.message;
  const first = data?.errors && Object.values(data.errors)[0];
  return Array.isArray(first) ? first[0] : fallback;
}

import { Navigate, Outlet, useLocation } from "react-router";
import { useAuth } from "@/hooks/useAuth";
import { ROUTES } from "@/routes/path";

function FullScreenLoader() {
  return <div className="grid min-h-screen place-items-center text-muted-foreground">Loading…</div>;
}

export function ProtectedRoute() {
  const { isAuthenticated, isLoading } = useAuth();
  const location = useLocation();
  if (isLoading) return <FullScreenLoader />;
  if (!isAuthenticated)
    return <Navigate to={ROUTES.auth.login} replace state={{ from: location }} />;
  return <Outlet />;
}

export function GuestRoute() {
  const { isAuthenticated, isLoading } = useAuth();
  const location = useLocation();
  if (isLoading) return <FullScreenLoader />;
  if (isAuthenticated) {
    const to = location.state?.from?.pathname ?? ROUTES.dashboard;
    return <Navigate to={to} replace />;
  }
  return <Outlet />;
}

16. End-to-end flows

Fresh login

Type creds → store login() → POST /admin/login → token+user into the store, status authenticated, cookie set → navigate to dashboard.

Reload on a protected page

App boots → status="loading" → guards show loader → effect calls bootstrap() → POST /admin/refresh (cookie) → token re-minted → page renders. No flash of login.

Access token expires mid-session

Request 401 → interceptor refreshes once → setToken → replays request (+ any queued) → user notices nothing.

Refresh token expired

Refresh 401 → interceptor calls store clear() → session reset + toast → guard redirects to /login.

Logout

Store logout() → POST /admin/logout (revokes token, clears cookie) → session reset → guard sends to /login.

Theme

toggle() flips <html class="dark">, persists to localStorage via persist, and Sonner re-themes via useTheme.
