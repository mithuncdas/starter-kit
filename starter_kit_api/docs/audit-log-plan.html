1. Why Chronicle (not spatie/activitylog)

The differences boil down to three things this project actually cares about:

Concernspatie/activityloglaravel-chronicle/core
Tamper detectionNone — rows are mutable.Append-only ledger; each row holds chain_hash = SHA256(prev_chain_hash + payload_hash). Any retroactive edit breaks the chain.
Cryptographic signingNone.Ed25519 signatures on checkpoints. Exports are signed and re-verifiable.
Export for compliance / forensicsDIY.php artisan chronicle:export emits entries.ndjson + manifest.json + signature.json that any third party can verify via chronicle:verify-export.
API ergonomicsactivity()->performedOn(...)->causedBy(...)Chronicle::record()->actor(...)->action(...)->subject(...)->metadata(...)->commit()
Eloquent auto-recordingLogsActivity traitHasChronicle trait (created/updated/deleted with diff)
Bottom line: if “could the DB admin quietly fix a row last week” is a question someone might ever ask, you want a hash-chained ledger, not a regular log table. Chronicle costs the same to set up; the integrity guarantees come free.

2. What Chronicle Gives Us

  • Append-only chronicle_entries table with ULID PKs, actor_type/id, action, subject_type/id, payload, payload_hash, chain_hash, checkpoint_id, metadata, tags, correlation_id, diff, context, created_at.
  • Sibling chronicle_checkpoints table that periodically anchors the chain head with a signature.
  • HasChronicle trait — drop on a model and created / updated / deleted events auto-record with a diff (timestamps automatically ignored).
  • Chronicle::record() facade — fluent builder for any custom event that doesn’t map cleanly to Eloquent (logins, OTP issued/verified, password reset, role sync, etc.).
  • Drivers: eloquent (default, synchronous), queued (async on a dedicated single-worker queue), array (tests), null (kill-switch).
  • Artisan: chronicle:install, chronicle:export, chronicle:verify-export.
  • Optional read-only Blade UI gated by CHRONICLE_UI_ENABLED=true — not used here; we’ll surface logs through the existing API.

4. Recording Strategy

One pattern across the whole system: every audit entry is written by an explicit Chronicle::record() call from a service class, inside the same DB transaction as the mutation it describes. No model traits. No event listeners.

The services that own each surface:

ServiceOwns the audit calls for
AdminAuthServiceLogin, logout, password reset, password change, profile update
OtpServiceOTP issued, verified, verify-failed
AdminUserServiceAdmin user create / update / delete; the role-change and status-change side entries
RoleServiceRole create / update / delete; the permission-sync side entry
UserAddressService (new — extracted from UserAddressController)Address create / update / delete; the primary-set side entry

Why this is the right single approach for this codebase:

  • Auth / OTP / password aren’t Eloquent writes — there’s no model event a trait could hook into. A trait alone is impossible here.
  • Splitting one user action into multiple semantic entries (e.g. an admin update producing admin_user.updated + admin_user.role_changed + admin_user.status_changed) requires control of the call site. A trait can only fire one generic updated event.
  • The project already has a service-per-feature convention (AdminAuthService, AdminUserService, RoleService, OtpService). Adding audit calls there keeps audit, business logic, and transactions co-located.
  • Sensitive-field scrubbing becomes obvious — the service chooses what enters payload / diff, so passwords and OTP values never accidentally leak through generic diff-of-dirty-attributes behavior.
Trade-off accepted: every CRUD service has to remember to call Chronicle::record(). The mitigation is a feature test per mutation path that asserts the entry exists (see §15) and a project rule that new write services land with their audit test.
UserAddressController refactor: the controller currently runs DB::transaction() blocks inline. To keep the audit pattern uniform, the transactions move into a new UserAddressService (sibling of AdminUserService under App\Services\Admin). The controller becomes a thin pass-through. See §8.6.

7. Action Naming Convention

Every action is dotted, snake_cased, and reads <feature>.<noun>.<verb_past_tense>. Three levels max. Verbs use the past tense because the entry is recorded after the change.

FeaturePatternExamples
Authauth.*auth.login.succeeded, auth.login.failed, auth.logout
Passwordauth.password.*auth.password.forgot_requested, auth.password.reset_completed, auth.password.changed
OTPotp.*otp.issued, otp.verified, otp.verify_failed, otp.exhausted
Admin useradmin_user.*admin_user.created, admin_user.updated, admin_user.deleted, admin_user.status_changed, admin_user.role_changed, admin_user.password_reset_by_admin
Rolerole.*role.created, role.updated, role.deleted, role.permissions_synced
Profileprofile.*profile.updated
Addressuser_address.*user_address.created, user_address.updated, user_address.deleted, user_address.primary_set

Standard tags we’ll attach so filters in the read API are fast:

  • auth — anything authentication-related.
  • security — failed logins, OTP failures, suspicious activity.
  • admin — mutations on admin users.
  • rbac — roles, permissions, role assignments.
  • profile — self-service profile changes.
  • address — address CRUD.

8. Per-Feature Audit Map

For each mutation, this table picks the service that owns it, the action key, the subject, and what goes into metadata / diff. Every entry is recorded via Chronicle::record() inside the service’s transaction (see §4). Implement the table top-to-bottom — that’s also the steps in §16.

8.1 Authentication

TriggerActionActorSubjectMetadata / Diff
Successful admin login auth.login.succeeded The user (just logged in) The user metadata: { ip, user_agent, device_name }; tags: [auth]
Failed login attempt auth.login.failed system (we don’t know who tried) The User by email if found, else null + email in metadata metadata: { email_attempted, reason: invalid_credentials|deactivated, ip, user_agent }; tags: [auth, security]
Logout auth.logout The user The user metadata: { token_id, ip }; tags: [auth]
All three are recorded from AdminAuthService::login() and the logout controller path. The throw branches inside login() emit auth.login.failed before re-throwing; the return path emits auth.login.succeeded. Same service, same transaction boundary.

8.2 Password & OTP

8.3 Admin users (CRUD)

Role swap inside same update (separate entry) admin_user.role_changed The User diff: { role: { old, new } }; tag admin, rbac

Why two entries for “update + role swap”? Because filtering on action = admin_user.role_changed across the whole ledger is the security team’s realistic query. Burying role changes inside a generic admin_user.updated diff hides them. Same goes for status flips on a deactivation.

8.7 No-trait rule

Across the entire codebase, no model uses HasChronicle. All audit calls are made explicitly from the responsible service. Specifically:

  • User, Role, UserAddress — audited via their services (AdminUserService, RoleService, UserAddressService).
  • Otp — audited via OtpService. The raw row contains the OTP hash and must never enter a generic model-event diff.
  • Permission, Country, AdminLevel, AdminArea, PostalCode, CountryAdminStructure — seeded reference data; no runtime writes, no audit needed.

Enforced by a project rule (CLAUDE.md or PR checklist): adding HasChronicle to any model is rejected in review.

9. Sensitive Field Handling

Three rules, applied everywhere:

  1. Never log the field value itself for: password, remember_token, otp (hashed or plaintext), plainTextToken from Sanctum.
  2. Log the fact that it changed, not the value. e.g. metadata: { password_changed: true } instead of diff: { password: { old, new } }.
  3. Email is logged because it’s already in users and showing up in diff doesn’t create new exposure. If GDPR scope grows later, pseudonymize at query time, not at write time — rewriting the ledger would break the hash chain.

How to implement:

Because every audit call is explicit (§4), the service simply doesn’t put the sensitive value into the payload / diff array it hands to EntryBuilder. EntryBuilder doesn’t scrub by default — the service controls exactly what goes in.

// AdminUserService::update() — correct:
Chronicle::record()
    ->actor($actor)
    ->action('admin_user.updated')
    ->subject($target)
    ->modelDiff($target)                              // diffs name/email/status
    ->metadata([
        'password_changed' => $passwordChanged,        // boolean only, never the value
        'tokens_revoked'   => $tokensRevoked,
    ])
    ->tags(['admin'])
    ->commit();

If a future refactor lets the service hand the whole model to modelDiff(), the model’s $hidden array (password, remember_token) protects against accidental serialization — but we still don’t rely on it; the rule is “construct the diff array yourself when sensitive fields are in play”.

10. Actor, Context & Correlation

10.1 Actor resolution

Chronicle defaults to Auth::user() ?? 'system'. That works for every authenticated admin endpoint. For:

  • Failed-login entries: no auth user, so actor is the literal string 'system' and the attempted email goes in metadata.
  • OTP issued during forgot-password: still no authenticated user (forgot-password is public). Actor = 'system'; subject = the looked-up User; metadata holds the requesting IP.
  • Queue jobs: actor = 'system' unless the job explicitly forwards the original user via a serialized field.

10.2 Correlation IDs

Wire a middleware that:

  1. Reads the incoming X-Correlation-Id header, or generates a ULID if absent.
  2. Sets it on Chronicle via the manager (or via the request context the package picks up).
  3. Echoes it back as a response header so the frontend can include it in error reports.

Result: every entry produced during a single API request shares the same correlation_id. Investigating “what happened in this request” becomes one query.

10.3 Context resolvers

Enable the bundled context resolvers in config/chronicle.php:

  • request — captures URL, method, IP, user agent.
  • environment — app env (production / staging).
  • queue — queue name + job class when recorded inside a job.

This way context on every entry is auto-populated; we don’t pass ip / user_agent by hand on each record() call.

11. Queueing & Performance

Start with eloquent (synchronous) in v1. Each audit entry adds one INSERT inside the request. For our load (admin panel, low write rate) this is invisible.

Promote to queued when either of these is true:

  • p95 of any audited endpoint creeps above ~50ms because of the audit write.
  • A single user action explodes into 3+ entries (current worst case is admin update → updated + role_changed + status_changed = 3 inserts, still fine).
If you switch to the queued driver, the chronicle queue MUST be processed by exactly one worker: 
php artisan queue:work --queue=chronicle --tries=1
Multiple workers race the chain-hash computation and produce forked chains, which breaks verification. The package documents this explicitly. Configure your supervisor / Horizon accordingly.

We don’t use the bundled Blade UI. Instead, expose Chronicle through the existing admin API surface.

12.2 Controller sketch

app/Http/Controllers/Admin/AuditLogController.php
use Chronicle\Entry\Entry;

public function index(IndexAuditLogRequest $request): JsonResponse
{
    $query = Entry::query();

    if ($request->filled('actor_type') && $request->filled('actor_id')) {
        $query->where('actor_type', $request->string('actor_type'))
              ->where('actor_id',   $request->string('actor_id'));
    }

    if ($request->filled('subject_type') && $request->filled('subject_id')) {
        $query->where('subject_type', $request->string('subject_type'))
              ->where('subject_id',   $request->string('subject_id'));
    }

    $entries = $query->orderByDesc('created_at')->cursorPaginate(50);

    return $this->success(
        data: AuditLogResource::collection($entries),
        message: 'Audit logs fetched.',
    );
}

public function show(string $entry): JsonResponse
{
    $row = Entry::query()->findOrFail($entry);

    return $this->success(
        data: AuditLogResource::make($row),
        message: 'Audit log fetched.',
    );
}

IndexAuditLogRequest rules:

'actor_type'   => ['nullable', 'string', 'required_with:actor_id'],
'actor_id'     => ['nullable', 'string', 'required_with:actor_type'],
'subject_type' => ['nullable', 'string', 'required_with:subject_id'],
'subject_id'   => ['nullable', 'string', 'required_with:subject_type'],

14. Retention, Export & Verification

14.1 Retention

365-day rolling window by default (config/chronicle.php → retention.days). The package ships a prune command; schedule it daily:

// routes/console.php
Schedule::command('chronicle:prune')->daily()->onOneServer();
Pruning is irreversible. Before raising or lowering the retention window, run an export of the soon-to-be-pruned range so the signed bundle is on cold storage. Treat the export as the “real” long-term record; the DB is the hot index.

14.2 Export

Manual or scheduled:

# All entries
php artisan chronicle:export

# Date-range bounded
php artisan chronicle:export --from=2026-01-01 --to=2026-03-31 --out=/storage/audit/q1-2026

Output is a directory with:

  • entries.ndjson — one JSON entry per line, in chain order.
  • manifest.json — counts, ranges, hash of entries.ndjson.
  • signature.json — Ed25519 signature of manifest.json.

14.3 Verification

php artisan chronicle:verify-export --path=/storage/audit/q1-2026

Reports: dataset hash match, signature validity, hash-chain integrity, range boundaries. Any failure means tampering or corruption between export and verification.

14.4 Scheduling

// routes/console.php
Schedule::command('chronicle:export --out=/storage/audit/monthly')
    ->monthlyOn(1, '02:00')
    ->onOneServer();

Then ship /storage/audit/monthly to S3 with bucket versioning + object-lock on. That’s the compliance copy.

15. Tests

The package ships a Chronicle::fake() facade method that swaps in an in-memory recorder, exposing assertions like assertRecorded(action: 'x', subject: $model). Use it for every audited path. Coverage we need:

  • AuthLoginAuditTest — successful login records auth.login.succeeded with actor=user, ip in metadata; failed login records auth.login.failed with email in metadata; deactivated login records auth.login.failed with reason=deactivated.
  • AuthLogoutAuditTest — auth.logout records with token_id.
  • PasswordFlowAuditTest — forgot → auth.password.forgot_requested, otp.issued; reset → otp.verified, auth.password.reset_completed; change → auth.password.changed. Verify no password value appears in any entry.
  • OtpFailureAuditTest — wrong code records otp.verify_failed with reason=invalid; max attempts records reason=exhausted.
  • AdminUserAuditTest — create/update/delete record the right actions; role swap inside an update produces an extra admin_user.role_changed entry; status flip produces an extra admin_user.status_changed entry; password changes never appear in diff.
  • RoleAuditTest — create/update/delete record the right actions; permission sync produces role.permissions_synced with added / removed arrays.
  • ProfileAuditTest — profile.updated diff has name/email only.
  • UserAddressAuditTest — UserAddressService::create/update/delete record the matching user_address.* entries; a write that demotes an existing primary produces an extra user_address.primary_set entry.
  • CorrelationIdAuditTest — multiple entries within one request share the same correlation_id; the header is echoed back.
  • SensitiveFieldsAuditTest — grep the recorded payloads for password, remember_token, otp — must be absent.
  • AuditApiTest — /admin/audit-logs filters work for actor pair and subject pair; /admin/audit-logs/{entry} returns the full entry; pagination works; audit.view gates both; without permission → 403; unknown entry id → 404.
  • ChainIntegrityTest — create 50 entries, run chronicle:export + chronicle:verify-export programmatically, expect green.

17. Risks & Open Questions

  • Queue worker discipline. The biggest operational footgun: switching to the queued driver without ensuring a single worker on the chronicle queue forks the hash chain. Document this in the runbook; lock the supervisor config; alert on multiple workers seen on the queue.
  • Private key handling. If CHRONICLE_SIGNING_PRIVATE_KEY leaks, an attacker can produce valid-looking signed exports. Treat like any production credential — secrets manager, rotation playbook, deploy-time injection only.
  • Retention vs compliance contracts. 365 days is a default, not a commitment. Check what the longest contract / regulation in scope demands (PCI: 1 year; SOX: 7 years; HIPAA: 6 years). Bump the env var accordingly; back-fill any pre-bump exports.
  • Failed-login enumeration via the audit log. The auth.login.failed entries include the attempted email. If audit.view is granted too liberally, that’s an enumeration surface. Keep audit.view on the Administrator role and any future explicit Auditor role; don’t hand it to general admins.
  • Limited filter surface. By design the read API only filters on actor + subject. Operators who want “all failed logins this week” or “all role syncs” will hit the DB directly or trigger a CLI export. If those questions become routine, revisit the filter list before each one gets bolted on ad-hoc.
  • Action-name sprawl. Three levels (feature.noun.verb) is a soft cap. Resist adding a fourth. If you find yourself wanting admin_user.password.reset_by_admin, prefer flattening to admin_user.password_reset_by_admin.
  • Performance under bulk imports. Today there are no bulk-import endpoints. If one ever ships (e.g. CSV import of admin users), wrap it in Chronicle::transaction() so the N entries get the same correlation id and aren’t lost to N individual flushes.
  • Forgetting an audit call. Because the approach is service-everywhere with explicit Chronicle::record() calls (§4), a new mutation method shipped without its audit call silently fails to log. Mitigations: every CRUD test asserts the audit entry exists (§15); PR review checklist requires the audit test for any new service method that writes.
  • Chronicle UI off by default. If someone flips CHRONICLE_UI_ENABLED=true in production, the bundled Blade routes appear unauthenticated unless the package’s middleware is also set. Keep it off; surface logs only through our gated API.
