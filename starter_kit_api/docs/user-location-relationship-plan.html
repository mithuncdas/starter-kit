1. Goals & Use Cases

The admin_areas table already gives us a recursive tree per country (for BD: Division → District → Upazila → Postal Code). The next step is to attach users to that tree so the app can answer:

  • Where does user #42 live? (full hierarchy)
  • List all users in Dhaka division / Demra upazila / postal code 1360.
  • Drive cascading address dropdowns on the user profile form.
  • Drive future delivery / serviceable-region / region-based pricing rules.

The design must be:

  • Country-agnostic — same model works for BD now, US/IN later.
  • Hierarchy-aware — storing the leaf (e.g. postal code) is enough; ancestors are derivable via the existing adjacency-list trait.
  • Extendable — today “the user’s location”; tomorrow billing/shipping/work addresses, lat/lon, delivery instructions.

2. Design Options & Trade-offs

OptionWhere the FK livesProsCons
A. Single FK column on users users.admin_area_id (nullable) → admin_areas.id Simplest. One join to get location. Ancestors derivable. One location per user. No room for street / lat-lon / label. Schema change to users every time location semantics grow.
B. Separate user_addresses table (1→many) user_addresses.admin_area_id Multiple addresses (home / work / billing / shipping). Street / lat-lon / label / primary flag. Doesn’t bloat users. Future-proof for orders. One extra table; one extra controller surface area.
C. Many-to-many pivot admin_area_user pivot Useful for “user serves regions” semantics. Wrong shape for an address — no place for street / lat-lon / per-address label.
Decision driver: the moment the product needs even one of {billing address, multiple addresses, latitude/longitude, delivery notes}, Option A becomes the wrong shape. Option B is the same complexity to build today and avoids a painful migration later.

3. Recommended Approach

Go with Option B — a dedicated user_addresses table. Each row points to a single admin_areas.id (typically the deepest available leaf for the country, e.g. POSTAL_CODE for BD), plus optional street / lat-lon / label / primary fields. A user can have N addresses; one is marked is_primary.

Country and intermediate ancestors (division/district/upazila) are derived from admin_area_id at read time via the HasRecursiveRelationships trait already on AdminArea. No duplication.

users ↑ (hasMany) user_addresses - admin_area_id ----> admin_areas (recursive tree) - is_primary - label (home | work | billing | shipping | other) - address_line1 / address_line2 - latitude / longitude - notes

4.3 UserAddressLabelEnum

app/Enums/UserAddressLabelEnum.php
namespace App\Enums;

enum UserAddressLabelEnum: int
{
    case Home     = 1;
    case Work     = 2;
    case Billing  = 3;
    case Shipping = 4;
    case Other    = 99;

    public function label(): string
    {
        return match ($this) {
            self::Home     => 'Home',
            self::Work     => 'Work',
            self::Billing  => 'Billing',
            self::Shipping => 'Shipping',
            self::Other    => 'Other',
        };
    }
}

5. Model & Relationships

5.1 UserAddress model

app/Models/UserAddress.php
namespace App\Models;

use App\Enums\UserAddressLabelEnum;
use Illuminate\Database\Eloquent\Attributes\Fillable;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;

#[Fillable([
    'user_id', 'admin_area_id', 'label', 'is_primary',
    'address_line1', 'address_line2', 'latitude', 'longitude', 'notes',
])]
class UserAddress extends Model
{
    protected function casts(): array
    {
        return [
            'label'      => UserAddressLabelEnum::class,
            'is_primary' => 'boolean',
            'latitude'   => 'decimal:7',
            'longitude'  => 'decimal:7',
        ];
    }

    /** @return BelongsTo<User, $this> */
    public function user(): BelongsTo
    {
        return $this->belongsTo(User::class);
    }

    /** @return BelongsTo<AdminArea, $this> */
    public function adminArea(): BelongsTo
    {
        return $this->belongsTo(AdminArea::class);
    }
}

5.2 Add relationships to User

app/Models/User.php (additions)
use Illuminate\Database\Eloquent\Relations\HasMany;
use Illuminate\Database\Eloquent\Relations\HasOne;

/** @return HasMany<UserAddress, $this> */
public function addresses(): HasMany
{
    return $this->hasMany(UserAddress::class);
}

/** @return HasOne<UserAddress, $this> */
public function primaryAddress(): HasOne
{
    return $this->hasOne(UserAddress::class)->where('is_primary', true);
}

5.3 Walking the hierarchy from an address

Because AdminArea uses HasRecursiveRelationships, getting the full chain is one call:

$user = User::with('primaryAddress.adminArea')->find($id);

// Full ancestry of that address (Division → District → Upazila → Postal Code)
$chain = $user->primaryAddress->adminArea
    ->ancestorsAndSelf()
    ->orderBy('depth')
    ->get();

6. Validation & Form Requests

6.1 StoreUserAddressRequest

app/Http/Requests/Admin/User/StoreUserAddressRequest.php
public function rules(): array
{
    return [
        'admin_area_id'  => ['required', 'integer', 'exists:admin_areas,id'],
        'label'          => ['required', new Enum(UserAddressLabelEnum::class)],
        'is_primary'     => ['boolean'],
        'address_line1'  => ['nullable', 'string', 'max:255'],
        'address_line2'  => ['nullable', 'string', 'max:255'],
        'latitude'       => ['nullable', 'numeric', 'between:-90,90'],
        'longitude'      => ['nullable', 'numeric', 'between:-180,180'],
        'notes'          => ['nullable', 'string'],
    ];
}

public function withValidator(Validator $validator): void
{
    $validator->after(function ($v): void {
        $areaId = (int) $this->input('admin_area_id');
        $area = AdminArea::query()->find($areaId);

        if ($area && ! $area->is_active) {
            $v->errors()->add('admin_area_id', 'This location is not active.');
        }
        // Optional: enforce leaf-level pick (deepest depth for the country)
        // $deepest = CountryAdminStructure::where('country_id', $area->country_id)->max('depth');
        // if ($area && $area->depth !== $deepest) {
        //     $v->errors()->add('admin_area_id', 'Pick a leaf-level location (e.g. postal code).');
        // }
    });
}
Choose the granularity policy up front:
  • Lenient — accept any depth. Useful when partial addresses are OK (“user is in Dhaka division”).
  • Strict — require the deepest level for that country (postal code for BD). Cleaner for reporting, harder for users without a postal code.
Default to lenient for v1; flip to strict by uncommenting the check above.

6.2 Primary-address uniqueness

One primary per user, enforced in the controller: on a save where is_primary=true, run UserAddress::where('user_id', $userId)->update(['is_primary' => false]) inside a transaction before saving the new row.

7. API Endpoints

Nested under the existing /admin/users module. Read access piggy-backs on admins.view; write access on admins.update (rationale: managing a user’s address is part of managing the user, not a separate concern). If you want a separate permission, see §8.

7.2 Resource shape

app/Http/Resources/Admin/UserAddressResource.php
public function toArray(Request $request): array
{
    return [
        'id'             => $this->id,
        'label'          => $this->label?->label(),
        'label_value'    => $this->label?->value,
        'is_primary'     => $this->is_primary,
        'address_line1'  => $this->address_line1,
        'address_line2'  => $this->address_line2,
        'latitude'       => $this->latitude,
        'longitude'      => $this->longitude,
        'notes'          => $this->notes,
        'admin_area'     => AdminAreaResource::make($this->whenLoaded('adminArea')),
        'hierarchy'      => $this->when(
            $this->relationLoaded('adminArea'),
            fn () => $this->adminArea?->ancestorsAndSelf()
                ->orderBy('depth')
                ->get()
                ->map(fn ($a) => ['level' => $a->level?->code, 'name' => $a->name])
                ->values(),
        ),
    ];
}

8. Permissions

Two paths:

  1. Re-use existing — admins.view / admins.update. Simpler, no new permissions to seed. Recommended for v1.
  2. Dedicated — add users.addresses.view, users.addresses.update in PermissionSeeder if address management needs to be delegated independently (e.g. a CSR role can edit addresses but not the user’s role/status).

9. Useful Queries

9.1 All users in a division

// $division is an AdminArea at depth=1
$userIds = UserAddress::query()
    ->whereIn('admin_area_id', $division->descendantsAndSelf()->select('id'))
    ->pluck('user_id')
    ->unique();

9.2 Users grouped by upazila

UserAddress::query()
    ->join('admin_areas', 'admin_areas.id', '=', 'user_addresses.admin_area_id')
    ->where('admin_areas.depth', 3) // upazila
    ->selectRaw('admin_areas.id, admin_areas.name, COUNT(DISTINCT user_addresses.user_id) AS user_count')
    ->groupBy('admin_areas.id', 'admin_areas.name')
    ->orderByDesc('user_count')
    ->get();

11. Step-by-Step Implementation Order

1

Migration

php artisan make:migration create_user_addresses_table. Paste the schema from §4.2. Run php artisan migrate.

2

Enum + Model + Factory

php artisan make:enum Enums/UserAddressLabelEnum --string=false (or hand-write). php artisan make:model UserAddress -mf — then replace the generated bodies with §5.1 and the factory from §10.

3

Wire User relationships

Add addresses() and primaryAddress() to App\Models\User (§5.2).

4

Form Requests

StoreUserAddressRequest, UpdateUserAddressRequest under App\Http\Requests\Admin\User. Identical rules; update can mark fields optional via sometimes.

5

Controller

App\Http\Controllers\Admin\UserAddressController with index/store/update/destroy. On save, if is_primary=true, unset other primaries in a transaction.

6

API Resource

UserAddressResource per §7.2, projecting label, hierarchy, and coords.

7

Routes

Nest under /admin/admin-users/{admin_user}/addresses with the existing controller-group pattern (§7.1).

8

Tests

Feature tests for permission gating + CRUD + primary-uniqueness. Run with php artisan test --compact tests/Feature/Admin/UserAddressTest.php.

9

Postman

Add a “User Addresses” folder under the Admin Users collection with the 4 requests; reuse {{auth_token}} bearer auth.

12. Future Extensions

  • Geo distance queries — once latitude/longitude are populated, add a nearby($lat, $lng, $kmRadius) scope on UserAddress using the Haversine formula (works on Postgres without PostGIS via raw SQL).
  • Address verification — integrate a postal API to verify postal_code ↔ upazila correctness on save.
  • Country switch — UI cascade: user picks country first, then division/district/upazila/postal-code dropdowns drive off the existing /admin/locations endpoints.
  • Soft deletes — if addresses are referenced by future tables (orders, invoices), add SoftDeletes on UserAddress so historical data stays intact.
  • Audit log — log address changes through an observer + the existing activity-log pipeline once that lands.
Builds on the existing Location Management plan. Read alongside that document — this plan touches no admin_areas schema changes, only adds user_addresses.
